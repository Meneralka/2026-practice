# MZD-42: Стратегия обработки сбоев и повторных вызовов

Для обеспечения надежности доставки сообщений используется паттерн **Transactional Outbox** и брокер Apache Kafka.

## 1. Гарантия доставки событий (Transactional Outbox)
* События пишутся в таблицу БД `outbox_table` в рамках одной транзакции с изменением статуса документа.
* Отдельный шедулер раз в секунду выгребает неотправленные записи и публикует их в топик `document.events.v1`.
* Состояние записи outbox — enum `OutboxStatus` (SPEC-13 §3.11): `NEW` → `IN_PROGRESS` → `SENT` | `FAILED` | `DEAD` (ушло в DLQ после исчерпания retry). Наружу этот статус не отдается.

## 2. Политика повторов (Retry Policy)
* **Для асинхронных событий (Kafka):** Допускается максимум 5 попыток отправки с экспоненциальной задержкой (backoff, стартовая — 2 с). При критическом сбое сообщение отправляется в Dead Letter Queue (DLQ) топик, запись outbox переводится в `DEAD`.
* **Для синхронных REST-вызовов:** Применяется политика Resilience4j Retry — максимум 3 попытки вызова с фиксированным интервалом в 1 секунду.

## 3. Недоставленные сообщения (DLQ / Manual Retry)
* Неотправленные события сохраняются в таблице `undelivered_messages` (TTL = 14 дней, синхронизировано с retention DLQ).
* Сообщения DLQ содержат только метаданные и идентификаторы (инвариант MZD-46 / MZD-40 §2): presigned-ссылок и бинарного содержимого в них нет, поэтому 14-дневное хранение не создает канала утечки файлов.
* Предоставляется REST API управления (`POST /api/v1/internal/outbox/retry-failed`) для ручного перезапуска отправки отбракованных транзакций администратором.
