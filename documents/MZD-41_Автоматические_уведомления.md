# MZD-41: Архитектура автоматических уведомлений

Вся рассылка в системе реализуется по событийно-ориентированной модели через брокер сообщений Apache Kafka. `document-service` только публикует события; непосредственную доставку до получателей выполняют специализированные сервисы-потребители.

## 1. Архитектурная схема
1. `document-service` фиксирует изменение статуса и публикует соответствующее событие в топик `document.events.v1` (через Transactional Outbox, MZD-42).
2. Внешние потребители вычитывают события и доставляют их адресатам:
   * **`cabinet-service`** — слушает топик и доставляет уведомления внешним заявителям в личный кабинет (ЛК).
   * **`workflow-service`** — слушает топик и распределяет уведомления / задачи среди внутренних пользователей (экспертов, координаторов, согласующих).

## 2. Матрица генерации уведомлений

| Исходное событие | Сервис-потребитель | Получатель и суть уведомления |
| :--- | :--- | :--- |
| `DocumentReplacementRequested` | `cabinet-service` | Заявитель (ЛК): «Требуется заменить файл в документе. Причина: {подпись reasonCode}» |
| `DocumentPublished` | `cabinet-service` | Заявитель (ЛК): «Документ {regNumber} утвержден и направлен в личный кабинет» |
| `DocumentTechnicalCheckFailed` | `cabinet-service` | Заявитель (ЛК), только для документов заявителя (`visibilityProfile = APPLICANT_REQUEST_MATERIAL`): «Файл не прошел проверку. Причина: {подпись reasonCode}» |
| `DocumentApprovalRequested` | `workflow-service` | Согласующий (АРМ): задача на согласование документа по маршруту |

## 3. Правила формирования текста
* В уведомления заявителю подставляется **русская подпись `reasonCode`** из словаря (`GET /internal/v1/dictionaries/ReasonCode`, SPEC-13 §0.5). Поле `comment` события помечено `internalOnly` и во внешние уведомления **не попадает**.
* Поле `regNumber` берется из payload события `DocumentPublished` (MZD-40).
* По документам типа `SERVICE_RESOLUTION` уведомления заявителю не формируются ни при каких условиях: события публикации для этого типа не существуют (guard MZD-22 §2.5), а прочие события резолюции `cabinet-service` обязан игнорировать.
