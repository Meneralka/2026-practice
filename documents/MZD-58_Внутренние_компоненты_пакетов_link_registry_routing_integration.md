# MZD-58 / SPEC-11: Внутренние компоненты пакетов link, registry, routing, integration

Настоящая спецификация описывает детальное устройство внутренних компонентов и сервисов для пакетов `link/`, `registry/`, `routing/` и `integration/` платформы управления документами. Спецификация опирается на архитектурные требования и интерфейсы, зафиксированные в MZD-21 и MZD-22.

---

## 1. Пакет `link/` (Компонент DocumentLink)

Пакет отвечает за установление и контроль связей между документами и смежными доменными сущностями системы.

### 1.1. Модель данных и кардинальность (M:N)
В соответствии со SPEC-13, связь документов и бизнес-объектов реализуется по схеме **Many-to-Many (M:N)**. Один документ может одновременно принадлежать нескольким сущностям, а одна бизнес-сущность может включать множество документов.

Связующая сущность `DocumentLink` описывается следующей структурой:
*   `documentId` (UUID) — Идентификатор карточки документа `DocumentCard`.
*   `targetId` (UUID) — Идентификатор целевого связанного бизнес-объекта.
*   `linkType` (Enum) — Тип связи, определяющий смысловую привязку. Использование устаревшего наименования `relatedEntityType` запрещено.

### 1.2. Допустимые типы связей (`linkType`)
Задан строгий список из 7 типов связей (согласно REQ-08-005):
1.  `REQUEST` — Заявка
2.  `EXPERTISE_CASE` — Дело
3.  `EXPERT_CARD` — Экспертная карта
4.  `REVIEW` — Проверка
5.  `PRIMARY_EXPERTISE` — Первичная экспертиза (необходима для сквозной кросс-документальной валидации)
6.  `REPORT` — Отчет
7.  `ROUTE` — Маршрут

## 2. Пакет `registry/` (Компонент нумераторов)

Пакет отвечает за гарантированное ведение регистрационных журналов и бессбойное выделение уникальных регистрационных номеров для документов.

### 2.1. Интерфейс генерации номера
Взаимодействие с нумератором осуществляется через следующие структуры:

*   **Входной DTO:** `GenerateNumberRequest`
    *   `documentType` (String) — Тип регистрируемого документа.
    *   `journalId` (UUID) — Идентификатор целевого регистрационного журнала.
    *   `idempotencyKey` (UUID / String) — Сквозной ключ идемпотентности, передаваемый клиентом.
    *   *Примечание:* Параметр «код подразделения» исключен согласно SPEC-16.
*   **Выходной DTO:** `RegistrationNumberResponse`
    *   `registrationNumber` (String) — Сформированный строковый регистрационный номер документа.
    *   `journalRecordId` (UUID) — Идентификатор созданной записи в журнале регистрации.

### 2.2. Механизм обеспечения идемпотентности
Идемпотентность выделения номеров контролируется непосредственно на стороне сервера:
1. При получении запроса с ранее обработанным ключом `idempotencyKey` в рамках одной пары (`documentType`, `journalId`), система не выполняет повторный инкремент счетчиков.
2. Сервер извлекает из БД или транзакционного кэша ранее сгенерированную запись и возвращает существующий номер в `RegistrationNumberResponse`.

## 3. Пакет `routing/` (Компоненты стейт-машин)

Пакет `routing/` отвечает за логику переходов документов по жизненным циклам. Он не хранит и не меняет статус напрямую, а взаимодействует с состоянием через внутренний интерфейс сервиса документов пакета `document/`.

### 3.1. Интерфейс выполнения переходов
*   **Входной DTO:** `ExecuteTaskCommand`
    *   `idempotencyKey` (UUID / String) — Ключ для исключения повторных вызовов.
    *   `taskId` (UUID) — Идентификатор текущей технологической или бизнес-задачи.
    *   `action` (String) — Совершаемое действие (например, `accept`, `approve`, `publish`).
*   **Механизм идемпотентности (ФТ-35):**
    *   Все обработанные ключи `idempotencyKey` фиксируются в таблице идемпотентности в БД.
    *   При дублировании запроса повторное выполнение бизнес-логики и смена статусов блокируются — система возвращает текущее актуальное состояние без изменений.
    *   Механизм JPA-оптимистической блокировки через аннотацию `@Version` применяется исключительно для предотвращения коллизий параллельной обработки (race conditions) на уровне БД, но не заменяет собой прикладную проверку идемпотентности.

### 3.2. Настройки стейт-машин для 5 типов документов
Каждый тип документа обрабатывается изолированным автоматом переходов:

#### 1. Материалы заявки (`REQUEST_MATERIAL`)
*   **Граф переходов:** `DRAFT` $\rightarrow$ `UNDER_REVIEW` $\rightarrow$ `ACCEPTED` $\rightarrow$ `ARCHIVED`, с возможным циклом `UNDER_REVIEW` $\rightarrow$ `REPLACEMENT_REQUIRED` $\rightarrow$ `UNDER_REVIEW`.
*   **Ограничение возврата:** При обнаружении ошибок из `UNDER_REVIEW` документ переводится в `REPLACEMENT_REQUIRED`. После загрузки новой версии (получившей статус `CLEAN`) документ возвращается строго в `UNDER_REVIEW`, минуя прямой переход в `ARCHIVED`.
*   **Исключения:** Статусы `UNDER_APPROVAL`, `APPROVED` и `PUBLISHED` исключены из конфигурации.

#### 2. Экспертная карта (`EXPERT_CARD`)
*   **Граф переходов:** `DRAFT` $\rightarrow$ `UNDER_REVIEW` $\rightarrow$ `ACCEPTED` $\rightarrow$ `UNDER_APPROVAL` $\rightarrow$ `APPROVED` $\rightarrow$ `ARCHIVED`.
*   **Ограничение возврата:** Статус `REPLACEMENT_REQUIRED` (требование замены) может быть инициирован со стадий `UNDER_REVIEW` и `UNDER_APPROVAL`.
*   **Исключения:** Статус `PUBLISHED` полностью исключен — публикация оригинальной экспертной карты заявителю невозможна.

#### 3. Экспертное заключение (`EXPERT_CONCLUSION`)
*   **Граф переходов:** `DRAFT` $\rightarrow$ `UNDER_REVIEW` $\rightarrow$ `ACCEPTED` $\rightarrow$ `UNDER_APPROVAL` $\rightarrow$ `APPROVED` $\rightarrow$ `PUBLISHED` $\rightarrow$ `ARCHIVED`.
*   **Особенность:** Переход в `ACCEPTED` является строго обязательным технологическим шагом общего шаблона и не может быть опущен.

#### 4. Сравнительный протокол (`COMPARATIVE_PROTOCOL`)
*   **Граф переходов:** `DRAFT` $\rightarrow$ `UNDER_REVIEW` $\rightarrow$ `ACCEPTED` $\rightarrow$ `UNDER_APPROVAL` $\rightarrow$ `APPROVED` $\rightarrow$ `ARCHIVED`, с возможными циклами `UNDER_REVIEW` $\rightarrow$ `REPLACEMENT_REQUIRED` $\rightarrow$ `UNDER_REVIEW` и `UNDER_APPROVAL` $\rightarrow$ `REPLACEMENT_REQUIRED` $\rightarrow$ `UNDER_APPROVAL`.
*   **Ограничение возврата:** Перевод в `REPLACEMENT_REQUIRED` доступен только из состояний `UNDER_REVIEW` и `UNDER_APPROVAL`.

#### 5. Служебная резолюция (`SERVICE_RESOLUTION`)
*   **Граф переходов:** `DRAFT` $\rightarrow$ `UNDER_APPROVAL` $\rightarrow$ `APPROVED` $\rightarrow$ `ARCHIVED`.
*   **Исключения:** Статусы `UNDER_REVIEW`, `ACCEPTED`, `PUBLISHED` и `REPLACEMENT_REQUIRED` запрещены.
*   **Безопасность (Guard):** На уровне стейт-машины действует жесткий `Guard`: любые попытки внешнего перевода или вызова неподдерживаемых методов возвращают ошибку `SecurityAccessDeniedException`.

## 4. Пакет `integration/` (Outbox-поллер)

Пакет реализует паттерн Transactional Outbox для надежного обмена сообщениями со смежными микросервисами через гарантированную публикацию событий в брокер.

### 4.1. Устройство outbox-поллера
*   **Запись событий:** Любое изменение состояния документа (загрузка версии, перевод статуса) фиксируется в рамках одной транзакции СУБД с записью события в системную таблицу `outbox_table`.
*   **Асинхронный опрос:** Фоновый процесс (поллер) считывает необработанные записи из `outbox_table`, отправляет их в брокер и при подтверждении успешной доставки (ACK) помечает события в таблице как обработанные.

### 4.2. Формат конверта события (Event Envelope)
Все публикуемые события сериализуются в JSON-конверт строго фиксированного формата:

```json
{
  "eventId": "UUID",
  "eventVersion": "1.0",
  "eventType": "DocumentPublished", 
  "occurredAt": "2026-07-17T10:15:00Z",
  "producer": "document-service",
  "causationId": "UUID-родительского-события-команды",
  "correlationId": "UUID-сквозного-процесса",
  "businessKey": "REQUEST-2026-000123 (ID заявки/бизнес-процесса, НЕ карточки документа)",
  "payload": {
    "fileId": "UUID",
    "mimeType": "application/pdf",
    "fileVersionId": "UUID"
  }
}
```
### 4.3. Базовый маппинг событий интеграционного контура
При переходах стейт-машины генерируются следующие события:
*   **Добавление и первичная загрузка новой версии файла - `DocumentUploaded`.**
*   **Переход карточки в статус `REPLACEMENT_REQUIRED` - `DocumentReplacementRequested`.**
*   **Переход карточки в статус `ACCEPTED` - `DocumentAccepted`.**
*   **Переход карточки в статус `APPROVED` - `DocumentApproved`.**
*   **Переход карточки в статус `PUBLISHED` - `DocumentPublished`.**
