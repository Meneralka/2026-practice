Перечень функциональных требований к document-service

Декомпозиция функциональности модуля загрузки документов и ЭДО с привязкой к требованиям ТЗ-08 (REQ-08-001 … REQ-08-012). Источники: ТЗ-08 (v1.0) и «Проект ответа об уточнении интеграций document-service» (v0.1).

Целевой сервисdocument-service (Java 21, Spring Boot 3.x, Maven, PostgreSQL + JPA, Flyway, Kafka, RustFS/S3, OAuth2 JWT)
Базовый документТЗ-08-DOCUMENTS-EDO, разделы 5, 6, 7, 8, 10, 11
Владельцы бэкенда@koshak1030 — пакеты document/, delivery/ · @ArtemKvvs — пакеты link/, registry/, routing/, integration/
СтатусРабочий перечень для планирования и приёмки. Требования, помеченные «выводимое», реконструированы из доменной логики / проекта интеграций и подлежат подтверждению у стейкхолдера ТЗ до реализации.

1. Условные обозначения

MUST — обязательное (по ТЗ) SHOULD — желательное (по ТЗ) выводимое — не в таблице REQ, следует из разделов ТЗ / проекта интеграций

2. Перечень ФТ и сопоставление с REQ-08-XXX

ФТ Функциональное требование REQ-08 Пр-т Влад./пакет Критерий приёмки (уточнённый)
Группа 1. Загрузка и приём файлов — пакет document/
ФТ-01 Множественная загрузка файлов через API (multipart) и UI REQ-08-001 MUST К / document/ За одну операцию принимается несколько файлов; по каждому создаётся карточка/версия; результат по каждому файлу возвращается отдельно, ошибка одного не откатывает остальные.
ФТ-02 Двухфазная загрузка: init (карточка загрузки, uploadId, параметры) → complete (фиксация версии) REQ-08-001 MUST К / document/ Endpoint uploads/init возвращает uploadId, documentId, target, maxSize, allowedMimeTypes; complete завершает загрузку и создаёт FileVersion.
ФТ-03 Определение MIME/типа через Apache Tika и валидация по whitelist REQ-08-002 MUST К / document/ Тип определяется по содержимому (Tika), а не по расширению; файл вне whitelist отклоняется с кодом причины; факт — в аудите.
ФТ-04 Валидация размера файла по лимиту REQ-08-002 MUST К / document/ Превышение maxSize отклоняется до сохранения в хранилище; сообщение пользователю функциональное, без техдеталей.
ФТ-05 Расчёт SHA-256 и контроль целостности REQ-08-002
REQ-08-004		 MUST К / document/ По каждой версии сохраняется sha256; повторная загрузка идентичного контента детектируется по хэшу.
ФТ-06 Антивирусная проверка через ClamAV (clamd) REQ-08-002 MUST К / document/ Каждый файл проходит clamd; заражённый не публикуется и не принимается; результат (passed/failed, reasonCode) фиксируется.
ФТ-07 Карантинная обработка при непрохождении проверки REQ-08-002
выводимое		 MUST К / document/ Файл, не прошедший тех/АВ-проверку, остаётся в статусе «Черновик»/карантин, связь с бизнес-объектом и статус «На проверке» не устанавливаются.
Группа 2. Карточка документа и версии — пакет document/
ФТ-08 Создание карточки документа (DocumentCard) с реквизитами REQ-08-003 MUST К / document/ Карточка содержит documentId, businessKey, documentType, status, versionNumber, actualFileVersionId, visibilityProfile; DTO отделены от сущностей.
ФТ-09 Получение карточки по id REQ-08-003 MUST К / document/ GET по documentId возвращает карточку, актуальную версию, статус, связи, признаки видимости и аудит по правам.
ФТ-10 Список карточек с фильтрами и пагинацией REQ-08-003 MUST К / document/ Фильтрация по businessKey, documentType, status, linkType, targetId, visibilityProfile; постраничная выдача.
ФТ-11 Тип-специфичные реквизиты для 5 типов документов REQ-08-003
выводимое		 MUST К / document/ Реквизиты типов (REQUEST_MATERIAL, EXPERT_CARD, EXPERT_CONCLUSION, COMPARATIVE_PROTOCOL, SERVICE_RESOLUTION) отображены на поля сущности. Разрыв в MZD-5: маппинг реквизитов к полям не выполнен — требует проработки.
ФТ-12 Версионирование без перезаписи: каждая загрузка — новая FileVersion REQ-08-004 MUST К / document/ Новая загрузка создаёт версию с versionNumber и previousVersionId; старый объект в хранилище не перезаписывается.
ФТ-13 Указание актуальной версии и её статуса REQ-08-004 MUST К / document/ actualFileVersionId однозначно указывает текущую версию. Разрыв MZD-5: FileVersion.status без enum, пересечение с checkResult/isCurrent — развести семантику.
ФТ-14 История версий документа REQ-08-004 MUST К / document/ Доступен список всех версий с датами, авторами, хэшами; старые версии читаемы по правам.
Группа 3. Хранилище, ссылки, предпросмотр — пакет document/
ФТ-15 Сохранение файла в объектное хранилище (RustFS/Minio через AWS SDK v2, S3 API) REQ-08-001
выводимое		 MUST К / document/ Файл кладётся по политике bucket/prefix; в FileVersion сохраняются objectStorageRef, storageBucket, storageKeyHash.
ФТ-16 Защищённые временные (presigned) ссылки на просмотр/скачивание с TTL REQ-08-009 SHOULD К / document/ createSecureLink(documentId, action=VIEW|DOWNLOAD, ttl) возвращает ссылку с expiresAt; действие фиксируется.
ФТ-17 Предпросмотр файла в браузере REQ-08-008 MUST К / document/ Поддерживаемые форматы открываются на просмотр без скачивания, по защищённой ссылке и правам.
ФТ-18 Неизменяемость утверждённых версий (object lock / WORM) REQ-08-004
выводимое		 SHOULD Об+К Утверждённые/опубликованные версии защищены от изменения на уровне хранилища (проверка фичи WORM — на @Hqhzx).
Группа 4. Публикация и доставка заявителю — пакет delivery/
ФТ-19 Публикация документа заявителю с профилем видимости REQ-08-010 MUST К / delivery/ publish(documentId, applicantId, visibilityProfile) переводит документ в «Направлен заявителю»; публикуется DocumentPublished.
ФТ-20 Проверка visibilityProfile при публикации REQ-08-010
выводимое		 MUST К / delivery/ Внутренние комментарии/служебные решения/закрытые материалы не попадают в кабинет; профиль видимости валидируется перед выдачей.
ФТ-21 Отображение результата в личном кабинете заявителя REQ-08-010 MUST К / delivery/ Заявитель видит опубликованный документ; getPublishedDocuments(applicantId, businessKey) отдаёт доступный перечень.
ФТ-22 Регистрация DeliveryReceipt: доставлен / открыт / скачан REQ-08-011 SHOULD К / delivery/ Фиксируются receiptType, occurredAt, ipHash, userAgentHash; document-service хранит факт как документарный, событие формирует cabinet-service.
Группа 5. Связи документа — пакет link/
ФТ-23 Создание связи документа с бизнес-объектом (заявка/дело/карта/отчёт/маршрут) REQ-08-005 MUST А / link/ createDocumentLink(documentId, linkType, targetId); связь создаётся идемпотентно, факт — в аудите и событии DocumentLinkCreated.
ФТ-24 Каталог типов связей + поле relatedEntityType REQ-08-005
выводимое		 MUST А / link/ linkType: REQUEST, EXPERTISE_CASE, EXPERT_CARD, REPORT, ROUTE, PRIMARY_EXPERTISE. Разрыв MZD-5/MZD-6: ввести явный relatedEntityType, устранить конфликт с relatedEntityType на DocumentCard.
ФТ-25 Кардинальность связи M:N REQ-08-005
выводимое		 MUST А / link/ Один документ может быть связан с несколькими объектами и наоборот. Разрыв MZD-6: M:N задать явно в модели.
ФТ-26 Получение связей документа REQ-08-005 MUST А / link/ GET связей по documentId и обратный поиск документов по targetId/linkType.
Группа 6. Реестры и нумерация — пакет registry/
ФТ-27 Генерация регистрационного номера документа REQ-08-006 MUST А / registry/ При регистрации присваивается номер по формату журнала; повтор по idempotencyKey не выдаёт новый номер.
ФТ-28 Регистрационные журналы REQ-08-006 MUST А / registry/ Журнал хранит последовательность номеров с датой, типом, автором; доступен для просмотра по правам.
ФТ-29 Гарантия уникальности номера (SELECT FOR UPDATE) REQ-08-006
выводимое		 MUST А / registry/ Конкурентная выдача номеров сериализуется, дублей нет. Открытый вопрос MZD-6: безразрывность (gapless) vs допущение пропусков — согласовать со стейкхолдером.
Группа 7. Маршруты ЭДО и статусная модель — пакет routing/
ФТ-30 Создание маршрута согласования (DocumentRoute) REQ-08-007 MUST А / routing/ startDocumentRoute(documentId, routeTemplateId, participants); маршрут переводит документ в «На согласовании» при наличии routeId.
ФТ-31 Задачи исполнителям (RouteTask) и фиксация решений REQ-08-007 MUST А / routing/ completeRouteTask(documentId, taskId, decision, comment); решения согласующих сохраняются с автором и временем.
ФТ-32 Движение статуса документа по маршруту REQ-08-007 MUST А / routing/ Положительное завершение → «Утверждён»; отклонение → «Требуется замена»; переходы только в рамках разрешённой матрицы.
ФТ-33 Тип-специфичные конечные автоматы статусов (5 типов × 8 статусов) REQ-08-007
выводимое		 MUST А / routing/ Применимость статусов различается по типам (матрица раздела 8 проекта интеграций), а не единый линейный ЖЦ. Повторяющийся разрыв — зафиксировать матрицу применимости и переходы.
ФТ-34 Протокол смены статуса DocumentCard через границу пакетов routing↔document REQ-08-007
выводимое		 MUST А+К Явный внутренний контракт изменения статуса карточки из routing. Разрыв MZD-6: протокол не определён — согласовать между владельцами пакетов.
ФТ-35 Идемпотентная команда смены статуса документа REQ-08-007 MUST А / routing/ POST status с idempotencyKey; повтор не меняет состояние второй раз; фиксируются previousStatus/newStatus/reasonCode.
Группа 8. События и интеграции — пакет integration/
ФТ-36 Публикация минимального набора событий (DocumentUploaded, DocumentAccepted, DocumentPublished) REQ-08-012 MUST А / integration/ События публикуются по согласованной схеме, содержат correlationId, корректно обрабатываются потребителем.
ФТ-37 Transactional Outbox: событие в одной транзакции с изменением данных, поллер → Kafka REQ-08-012 MUST А / integration/ При успешной транзакции событие не теряется; publish идёт только через outbox.
ФТ-38 Единый envelope: eventId, eventType, eventVersion, correlationId, causationId, businessKey REQ-08-012 MUST А / integration/ Все события содержат единый конверт и версию схемы; схема публикуется в реестре (AsyncAPI/Schema Registry).
ФТ-39 Явное отображение переходов статусов → события REQ-08-012
выводимое		 MUST А / integration/ Задан маппинг «переход статуса → событие», а не публикация на каждое изменение. Принцип: без blanket fire-on-every-change.
ФТ-40 Расширенный каталог событий (TechnicalCheckPassed/Failed, VersionCreated, LinkCreated, ApprovalRequested, RejectedByApprover, PublicationFailed, DeliveryReceiptRegistered, Archived) REQ-08-012
выводимое		 SHOULD А / integration/ Каталог из раздела 6 проекта интеграций. Требует подтверждения стейкхолдера ТЗ — перечень базового ТЗ не исчерпывающий.
ФТ-41 Потребление входящих событий соседних сервисов (consumer) REQ-08-012
выводимое		 MUST А / integration/ Consumer-ы для request/expertise/review/report/cabinet/workflow. Разрыв MZD-6: inbound-слой полностью отсутствовал.
ФТ-42 Вызовы соседних сервисов через WebClient/Feign (проверка контекста) REQ-08-012
выводимое		 MUST А / integration/ validate*Context(...) к соседям с timeout/retry/circuit breaker; прямой доступ к чужой БД запрещён.
ФТ-43 Идемпотентность consumer-ов (dedup по eventId) REQ-08-012
выводимое		 MUST А / integration/ Обработанные eventId хранятся; дубль не приводит к повторному действию.
ФТ-44 DLQ, consumer retry и ручной replay REQ-08-012
выводимое		 SHOULD А+Об После исчерпания retry сообщение уходит в DLQ с диагностикой; поддержан ручной replay (раздел 11 ТЗ: retry/outbox/DLQ).
Группа 9. Сквозные требования — разделы 3, 9, 10, 11 ТЗ
ФТ-45 Журнал аудита всех изменений разд. 10
выводимое		 MUST Об Изменения статусов, реквизитов, файлов, маршрутов и прав фиксируются с автором и временем (входит в DoD каждой задачи).
ФТ-46 Ролевая и атрибутная модель доступа разд. 3, 10
выводимое		 MUST Об Доступ по роли и атрибутам (организация, статус процесса, принадлежность к делу); проверка на уровне сервиса.
ФТ-47 Проверка токенов JWT (OAuth2 Resource Server), защита на gateway и сервисе разд. 10
выводимое		 MUST Об Все межсервисные и пользовательские вызовы — по защищённому каналу с проверкой токена и прав.
ФТ-48 Поиск / фильтрация / сортировка / пагинация / сохранённые представления разд. 9
выводимое		 MUST Об (@NITF1S) UI по OpenAPI-контракту бэкенда; списки поддерживают пагинацию и массовые действия в пределах прав.
ФТ-49 Наблюдаемость: health/readiness, метрики, трейсинг разд. 11
выводимое		 MUST Об (@Hqhzx) Actuator health/readiness, Micrometer+Prometheus, OpenTelemetry; в логах и трассировке — correlationId.
ФТ-50 Отказоустойчивость: retry, outbox, DLQ, отсутствие потери данных разд. 11
выводимое		 MUST Об+А Сбой внешнего сервиса/потребителя не приводит к потере данных; целевые настройки timeout/retry/circuit breaker зафиксированы.

4. Открытые вопросы (согласовать до реализации)

Требования с меткой «выводимое» и позиции ниже реконструированы из доменной логики и проекта интеграций v0.1; раздел 8 ТЗ по обмену задан шаблонной формулировкой и подлежит формальному уточнению у стейкхолдера ТЗ:
  • Нумерация (ФТ-29): безразрывная (gapless) или с допустимыми пропусками.
  • Матрица применимости статусов (ФТ-33): охват «На проверке» (только комплектность координатора или также тех/АВ-проверка); статусы для «служебной резолюции»; отдельный маршрут «экспертной карты»; применимость «Направлен заявителю» за пределами итогового заключения.
  • Расширенный каталог событий (ФТ-40) и содержание межсервисного обмена (ФТ-41, ФТ-42): подтвердить перечень, инициатора, направление и состав данных по каждому смежному сервису.
  • Границы области: не втягивать в document-service содержательную часть, принадлежащую другим сервисам (экспертные выводы, комплектность как расчёт, маршрут как сущность workflow).
