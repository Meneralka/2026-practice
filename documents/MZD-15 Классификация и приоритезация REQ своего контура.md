## 2. Кандидаты в «Перспективу» (Out of Scope / Future Backlog)

Ниже перечислены функциональные модули, которые вынесены из текущего релиза в бэклог развития платформы с целью снижения архитектурной сложности стартового контура.

### 2.1. Дедупликация файлов на основе контентного хэша
*   **Суть:** Проверка наличия файла в S3-хранилище по хэшу `sha256` до фактического выделения физического места, с организацией нескольких ссылок `FileVersion` на один физический S3-объект.[cite: 6]
*   **Обоснование переноса:** Требует реализации сквозного подсчета хэша на клиенте до загрузки файла, а также усложняет логику удаления/архивации объектов (необходим подсчет ссылок `Reference Counting` на уровне S3-адаптера). Риск коллизий при одновременной очистке данных.

### 2.2. Расширенные форматы интерактивного предпросмотра (Preview Engine)
*   **Суть:** Генерация легковесных PDF-, веб- или растровых копий для тяжелых чертежей, BIM-моделей или специфических форматов отчетности для их рендеринга непосредственно в интерфейсе `cabinet-service` или АРМ Эксперта.
*   **Обоснование переноса:** На текущем этапе `document-service` оперирует исключительно `presignedUrl`/`secureLinkRef` для скачивания оригинальных бинарных файлов силами браузера клиента.[cite: 5, 6] Внедрение конвертера требует создания изолированного ресурсоемкого пула воркеров.

### 2.3. Долгосрочное архивное хранение с отложенным «холодным» сжатием (Glacier Tiering)
*   **Суть:** Автоматический перенос файлов документов со статусом `Архив` в холодные классы хранения S3 (например, AWS Glacier / Yandex Cold Storage) по истечении 90 дней.[cite: 5, 6]
*   **Обоснование переноса:** На этапе запуска объемы хранилища не создают критической нагрузки по стоимости. Логика восстановления из "холодного" архива асинхронна и занимает от нескольких минут до часов, что требует переработки UI и UX ожидания ссылки пользователем.
