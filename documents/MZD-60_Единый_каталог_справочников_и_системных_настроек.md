# MZD-60: Единый каталог справочников и системных настроек

Сводный реестр системных перечислений (enums) и глобальных конфигураций для обеспечения работы контрактных тестов.

## 1. Системные перечисления (Enums)

### Типы документов (`documentType`)
* `REQUEST_MATERIAL` — Материалы заявки
* `EXPERT_CARD` — Экспертная карта (каноничное имя)
* `EXPERT_CONCLUSION` — Экспертное заключение
* `COMPARATIVE_PROTOCOL` — Сравнительный протокол
* `SERVICE_RESOLUTION` — Служебная резолюция

### Типы связей (`linkType`)
* `REQUEST` — Заявка
* `EXPERTISE_CASE` — Дело (без изменений)
* `EXPERT_CARD` — Экспертная карта (заменено согласно MZD-21)
* `REVIEW` — Проверка
* `PRIMARY_EXPERTISE` — Первичная экспертиза
* `REPORT` — Отчет
* `ROUTE` — Маршрут

### Статусы документов (`status`)
* `DRAFT` — Черновик
* `UNDER_REVIEW` — На проверке
* `ACCEPTED` — Принят
* `UNDER_APPROVAL` — На согласовании
* `APPROVED` — Утвержден
* `PUBLISHED` — Опубликован
* `REPLACEMENT_REQUIRED` — Требуется замена
* `ARCHIVED` — В архиве

### Профили видимости (`visibilityProfile`)
* `INTERNAL` — Только внутренний контур (единственно допустимый для `SERVICE_RESOLUTION`)
* `APPLICANT_RESULT` — Результат заявителю (`EXPERT_CONCLUSION` после публикации)
* `APPLICANT_REQUEST_MATERIAL` — Заявитель видит собственные загруженные материалы заявки

### Типы квитанций (`receiptType`)
* `DELIVERED` — Документ отобразился в реестре Личного Кабинета заявителя
* `OPENED` — Документ открыт заявителем в предпросмотре
* `DOWNLOADED` — Документ скачан заявителем

## 2. Справочник причин возврата и сбоев (`reasonCode`)

Коды приведены к единому справочнику `ReasonCode` (SPEC-13 §3.3, без префиксов `ERR_`/`REJ_`).

| Технический код (reasonCode) | Бизнес-причина | Применяется в событиях |
| :--- | :--- | :--- |
| **`FILE_CORRUPTED`** | Загруженный файл поврежден или не читается | `DocumentTechnicalCheckFailed` |
| **`ANTIVIRUS_INFECTED`** | Файл не прошел антивирусную проверку | `DocumentTechnicalCheckFailed` |
| **`RETURNED_FOR_REVISION`** | Отклонено согласующим лицом (требуется доработка) | `DocumentReplacementRequested`, `DocumentRejectedByApprover` |
| **`DATA_MISMATCH`** | Неверно заполнены атрибуты карточки документа | `DocumentReplacementRequested` |
| **`CABINET_UNAVAILABLE`** | Сервис личного кабинета недоступен по таймауту | `DocumentPublicationFailed` |

*Примечание:* код `ERR_SIGNATURE_INVALID` (проверка электронной подписи) исключен — он не входит в справочник `ReasonCode` SPEC-13 §3.3 и не подтвержден ни одним источником в корпусе. Пайплайн валидации (MZD-61 §1) проверки ЭП не выполняет.

## 3. Таблица системных настроек и лимитов

| Параметр настройки | Значение | Описание |
| :--- | :--- | :--- |
| **`link_ttl_staff`** | 15 минут | TTL защищенной ссылки для сотрудников ведомства (АРМ), рассчитывается сервером (BR-06, MZD-18) |
| **`link_ttl_applicant`** | 30 минут | TTL защищенной ссылки для заявителя (ЛК), рассчитывается сервером (BR-06, MZD-18) |
| **`file_size_limit`** | 50 МБ | Единый максимальный размер файла для всех типов документов (SPEC-04 A3) |
| **`kafka_retry_attempts`**| 5 | Максимальное количество автоматических повторов отправки в Kafka |
| **`rest_retry_attempts`** | 3 | Максимальное количество повторов для синхронных REST-запросов |
| **`dlq_retention_days`**  | 14 дней | Время хранения (TTL) записей в Dead Letter Queue / таблице ошибок |

*Примечание:* значения `link_ttl_draft` (15 мин) / `link_ttl_final` (60 мин) и раздельные лимиты `file_size_limit_low`/`file_size_limit_high` (50/250 МБ) исключены — они не подтверждены ни одним источником в корпусе. TTL зависит от роли пользователя, а не от статуса документа (BR-06); лимит размера файла — единый 50 МБ (SPEC-04 A3), без деления по типам документов.
