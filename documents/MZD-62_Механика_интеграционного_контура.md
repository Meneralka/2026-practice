## 1. Transactional Outbox

### 1.1. Размещение

| Параметр | Значение | Источник |
|---|---|---|
| Пакет | `integration/` | MZD-19 §1 (REQ-08-012) |
| Требование | REQ-08-012 · **MUST** | MZD-19 §1 |
| Ответственный | @ArtemKvvs (`link/` + `registry/` + `routing/` + `integration/`) | SPEC-04 |
| Продюсер | `document-service` | MZD-20, поле `producer` конверта |

### 1.2. Механика

```
┌─ Транзакция БД ──────────────────────────────┐
│  1. Изменение бизнес-состояния               │
│     (DocumentCard / FileVersion /            │
│      DocumentLink / статус)                  │
│                                              │
│  2. INSERT в таблицу outbox                  │
│     (событие в конверте SPEC-14)             │
│                                              │
│  COMMIT — атомарно оба действия              │
└──────────────────────────────────────────────┘
                     │
                     ▼
        ┌─ Outbox-поллер (отдельный поток) ─┐
        │  3. SELECT неопубликованных       │
        │  4. Publish в document.events.v1  │
        │  5. Пометка как опубликованных    │
        └───────────────────────────────────┘
```

**Инвариант (стейкхолдер-документ р.10):** потеря события при успешной транзакции изменения документа **недопустима**. Обратное тоже: при откате транзакции запись в outbox не коммитится → событие не публикуется (фантомов нет).

**Гарантия доставки:** at-least-once. Дубли при повторной публикации нейтрализуются дедупликацией на стороне потребителя (см. 4.1).

> ⚠️ **Не зафиксировано в MZD-19/20/21/23:** интервал поллинга, размер батча, стратегия очистки опубликованных записей outbox, поведение при недоступности Kafka во время поллинга. См. раздел 6.

### 1.3. Транзакционная связка с нумератором

Номер регистрации резервируется через `Sequence` PostgreSQL **до коммита** транзакции; при откате номер сгорает (MZD-23 §1). Outbox-запись коммитится той же транзакцией → событие с уже присвоенным номером не может быть опубликовано для откатившейся регистрации.

## 2. Асинхронный скан ClamAV

### 2.1. Модель состояний

Проверка на вирусы вынесена на уровень **`FileVersion.status`** — отдельно от статуса карточки (MZD-22 §1).

| Уровень | Поле | Значения | Источник |
|---|---|---|---|
| Файл | `FileVersion.status` | `CHECKING` → `CLEAN` \| `INFECTED` | MZD-22 §1, SPEC-13 |
| Карточка | `DocumentCard.status` | `DRAFT` на время проверок | MZD-22 §1 |

### 2.2. Механика

```
Загрузка файла (uploads/complete)
        │
        ▼
FileVersion.status = CHECKING
DocumentCard.status = DRAFT          ◄── карточка «заперта» на время скана
        │
        ▼
   Асинхронный скан ClamAV
        │
        ├── чисто ────► FileVersion.status = CLEAN
        │               событие DocumentTechnicalCheckPassed
        │               │
        │               └─► бизнес-переходы РАЗРЕШЕНЫ
        │                   (DRAFT → UNDER_REVIEW и далее по графу типа)
        │
        └── заражён ──► FileVersion.status = INFECTED
                        событие DocumentTechnicalCheckFailed
                        │
                        └─► карточка ОСТАЁТСЯ в DRAFT
                            связи (DocumentLink) НЕ создаются
                            бизнес-переходы ЗАБЛОКИРОВАНЫ
```

### 2.3. Правило допуска

> **Переход на бизнес-стадии разрешён только для файлов в статусе `CLEAN`** (MZD-22 §1).

Это правило перекрывает граф любого типа документа: даже там, где переход разрешён стейт-машиной, файл не в `CLEAN` его блокирует.

| Состояние файла | Бизнес-переход | Создание связи |
|---|---|---|
| `CHECKING` | заблокирован | заблокировано |
| `CLEAN` | разрешён по графу типа | разрешено |
| `INFECTED` | заблокирован (карантин) | заблокировано |

### 2.4. Асинхронность

Скан выполняется вне синхронного контура загрузки: `complete` возвращает ответ, не дожидаясь результата ClamAV. Результат приходит событием (`DocumentTechnicalCheckPassed` / `Failed`).

> ⚠️ **Не зафиксировано:** таймаут скана, поведение при недоступности ClamAV (файл вечно в `CHECKING`?), повторный скан, TTL карантина для `INFECTED`-файлов. См. раздел 6.

## 3. Каталог событий (16 системных)

**Конверт:** единый по **SPEC-14**, все 16 событий обёрнуты одинаково (MZD-21 §3).

```json
{
  "eventId": "UUID",
  "eventVersion": "1.0",
  "eventType": "DocumentUploaded",
  "occurredAt": "2026-07-12T10:15:00Z",
  "producer": "document-service",
  "causationId": "UUID-родительского-события-команды",
  "correlationId": "UUID-сквозного-процесса",
  "businessKey": "REQUEST-2026-000123 (ID заявки/бизнес-процесса, НЕ карточки документа — стейкхолдер-документ §2.1, §6.1)",
  "payload": { }
}
```

**Топик продюсера:** `document.events.v1` (топик-на-сервис-продюсер).

### 3.1. Полный каталог

| # | Событие | Когда публикуется | Потребители (MZD-20) |
|---|---|---|---|
| 1 | `DocumentUploadInitiated` | Создана карточка загрузки, выдан `uploadId` | — |
| 2 | `DocumentUploaded` | Файл загружен, создана версия (физическое добавление версии — MZD-22 §3) | request-service |
| 3 | `DocumentTechnicalCheckPassed` | `FileVersion.status = CLEAN` | — |
| 4 | `DocumentTechnicalCheckFailed` | `FileVersion.status = INFECTED` | — |
| 5 | `DocumentLinkCreated` | Документ связан с бизнес-объектом | request-service |
| 6 | `DocumentVersionCreated` | Создана новая версия без перезаписи | — |
| 7 | `DocumentAccepted` | Переход в `ACCEPTED` (MZD-22 §3) | request-service |
| 8 | `DocumentReplacementRequested` | Переход в `REPLACEMENT_REQUIRED` (MZD-22 §3) | review-service |
| 9 | `DocumentApprovalRequested` | Переход в `UNDER_APPROVAL` — инициация внешней оркестрации | workflow-service |
| 10 | `DocumentStatusChanged` | Изменён статус документа | workflow-service |
| 11 | `DocumentApproved` | Переход в `APPROVED` (MZD-22 §3) | expertise-service, review-service, report-service |
| 12 | `DocumentRejectedByApprover` | Согласующий отклонил документ | — |
| 13 | `DocumentPublished` | Переход в `PUBLISHED` (MZD-22 §3) — транзитное событие для ЛК | cabinet-service, report-service |
| 14 | `DocumentPublicationFailed` | Сбой доставки/публикации | report-service |
| 15 | `DocumentDeliveryReceiptRegistered` | Зарегистрирован факт доставки/открытия/скачивания | — |
| 16 | `DocumentArchived` | Переход в `ARCHIVED` | — |

**Итого: 16 событий** — соответствует MZD-21 §3 («расширяется до 16 системных событий»).

### 3.2. Семантический маппинг «переход → событие» (MZD-22 §3)

| Переход стейт-машины | Событие |
|---|---|
| Физическое добавление версии файла | `DocumentUploaded` |
| → `REPLACEMENT_REQUIRED` | `DocumentReplacementRequested` |
| → `ACCEPTED` | `DocumentAccepted` |
| → `APPROVED` | `DocumentApproved` |
| → `PUBLISHED` | `DocumentPublished` |

### 3.3. Ограничения содержимого

| Правило | Источник |
|---|---|
| Binary content в событиях **запрещён** | Стейкхолдер-документ §6.1 |
| Файлы передаются только ссылкой: `objectStorageRef` / `secureLinkRef` / `fileVersionId` | Стейкхолдер-документ §2 |
| Совместимые изменения — добавление optional-полей; breaking — новая major-версия | Стейкхолдер-документ §6.1 |

## 4. Consumer retry / DLQ / replay

### 4.1. Дедупликация входящих (закрыто — MZD-20)

> **Раздел закрыт** (MZD-20, «Фиксированные архитектурные правила»).

| Параметр | Значение |
|---|---|
| Ключ дедупликации | `eventId` из конверта SPEC-14 |
| Механизм | Таблица обработанных сообщений |
| Транзакционность | Обновление таблицы **в единой транзакции с бизнес-логикой** |
| Гарантия | Повторная доставка не приводит к повторному действию |

Это и есть защита при replay: сообщение можно переотправлять безопасно.

### 4.2. Идемпотентность команд (REST)

| Пакет | Механизм | Источник |
|---|---|---|
| `registry/` | Повтор с тем же `idempotencyKey` в паре (`documentType`, `journalId`) → сохранённый номер из кэша/БД, **без повторного инкремента** | MZD-21 §2, MZD-23 §1 |
| `routing/` | Таблица обработанных ключей идемпотентности (**ФТ-35**); при дубле повторная смена статуса блокируется, возвращается актуальное состояние | MZD-21 §2 |

> **Важное разграничение (MZD-21 §2):** `@Version` (JPA) — оптимистическая блокировка от конкурентных гонок, **не заменяет** функциональную идемпотентность. Это разные механизмы.

### 4.3. Retry-политика (НФТ, MZD-23 §2)

| Контур | Попыток | Задержка | Условие |
|---|---|---|---|
| **Kafka Consumer** | **5** | Экспоненциальная (backoff) | После исчерпания → карантинный топик (DLQ) |
| **REST-клиент (WebClient)** | **3** | — | При сетевых сбоях: 5xx, ConnectTimeout |

### 4.4. DLQ

| Параметр | Значение | Источник |
|---|---|---|
| **Топик** | `document-service-dlq` | MZD-23 §2 |
| **Retention** | **≥ 14 дней**, гарантированное сохранение в СУБД/хранилище | MZD-23 §2 |
| **Назначение** | Ручной разбор, исправление и повторный запуск (replay) администраторами | MZD-23 §2 |
| **Состав сообщения** | Исходное сообщение + `errorClass`, `errorMessage`, `stackTraceHash`, `consumer`, `failedAt`, `retryCount`, `correlationId` | Стейкхолдер-документ р.10 |

> ℹ️ **Закрывает ранее открытый вопрос:** DLQ — **единый топик на сервис** (`document-service-dlq`), а не `<topic>.DLT` на каждый исходный топик.

### 4.5. Replay

| Параметр | Значение |
|---|---|
| Режим | Ручной, администратором системы (MZD-23 §2) |
| Защита от дублей | Дедупликация по `eventId` в единой транзакции (4.1) |
| Порядок | В пределах одного ключа партиционирования — по `occurredAt` |

Детальный регламент разбора — в отдельном документе (runbook DLQ).

### 4.6. Delayed retry topic

Стейкхолдер-документ р.10 допускает delayed-retry-топик для временных ошибок. **В MZD-19/20/21/23 не упомянут** — условия перехода в него vs сразу в DLQ не зафиксированы. См. раздел 6.

## 5. Ключ партиционирования — ЗАКРЫТО стейкхолдер-документом v0.1

> ✅ **Вопрос закрыт.** «Проект ответа на запрос об уточнении интеграций document-service» v0.1 (стейкхолдер-документ, разделы §2.1 и §6.1) даёт прямой ответ, MZD-21 §3 в этой части был неточен.

**Определение `businessKey` (стейкхолдер-документ §2.1, пример конверта):**

```json
"businessKey": "REQUEST-2026-000123"
```

— это идентификатор **бизнес-процесса (заявки)**, а не карточки документа. Подтверждается таблицей «Идентификаторы процесса» (§7 стейкхолдер-документа): `businessKey` сгруппирован с `requestId`, `requestNumber` — отдельно от группы «Идентификаторы документа» (`documentId`, `documentType`, …). Формулировка в MZD-21 §3 (`"ID-ключ-карточки-документа"`) была неточной и исправлена (см. также правку в самом MZD-21).

**Ключ партиционирования (стейкхолдер-документ §6.1, дословно):**

| Тип события | Ключ |
|---|---|
| События процесса | `businessKey` |
| События конкретного документа | `documentId` |
| События публикации | `applicantId` (допускается, если нужно партиционирование по заявителю) |

Это правило уже было верно отражено в MZD-40 §1 («`documentId` для событий документа, `businessKey` для процессных») — расхождение существовало только в примере конверта MZD-21, не в правиле партиционирования.

**Статус:** число партиций и фактор репликации по-прежнему не финализированы (техническая настройка инфраструктуры, вне области стейкхолдер-документа) — см. раздел 6.2.
