# MZD-18 / SPEC-04: Бизнес-правила домена документов и доставки

## 1. Реестр бизнес-правил домена (Domain Business Rules)

### BR-01: Двухфазная загрузка файлов (Two-Phase Upload)
*   **Описание:** Процесс загрузки физического файла жестко разделен на два этапа. На первом этапе (`uploads/init`) сервис регистрирует метаданные карточки и выдает клиенту временную целевую ссылку для загрузки[cite: 6]. На втором этапе (`uploads/complete`) клиент подтверждает окончание отправки бинарного тела файла в S3[cite: 5, 6]. Прямая бесконтрольная запись в S3 в обход `document-service` заблокирована.
*   **Привязка к требованиям:** REQ-08-001[cite: 6].
*   **Критерий приемки (Вход в SPEC-11):** Попытка загрузить файл без предварительного вызова `/init` возвращает ошибку `403 Forbidden`. Карточка документа не создается в БД до успешной валидации метаданных на первой фазе[cite: 6].

### BR-02: Техническая валидация контента через Whitelist Apache Tika
*   **Описание:** Определение реального формата файла происходит на основе анализа его бинарного заголовка (`magic bytes`) библиотекой Apache Tika, а не по заявленному пользователем расширению[cite: 6]. Файл допускается к дальнейшей обработке только при совпадении с белым списком разрешенных MIME-типов для данного `documentType`[cite: 6].
*   **Привязка к требованиям:** REQ-08-002[cite: 6].
*   **Критерий приемки (Вход в SPEC-11):** Файл с измененным вручную расширением (например, переименованный `.exe` в `.pdf`) успешно загружается в S3, но после конвейера Tika переводится в `FileVersion.status` = `INVALID` и блокируется для бизнес-логики[cite: 6].

### BR-03: Жесткий лимит размера файла (Max Size Boundary)
*   **Описание:** Для каждого типа документа (`documentType`) в конфигурационной матрице системы задается максимальный объем в байтах[cite: 6]. Для исключения обхода лимитов при прямой загрузке по presigned-ссылке, ограничение проверяется дважды: превентивно на фазе `/init` через включение параметра `content-length-range` в POST Policy, а также на фазе `/complete` посредством синхронного HEAD-запроса `document-service` к бакету S3 для валидации фактического размера объекта.
*   **Привязка к требованиям:** REQ-08-002[cite: 5, 6].
*   **Критерий приемки (Вход в SPEC-11):** Если размер передаваемого файла превышает лимит хотя бы на 1 байт, вызов `/init` (или S3-загрузка по полиси), а также финальная HEAD-проверка на шаге `/complete` падают с ошибкой `413 Payload Too Large`[cite: 6].

### BR-04: Карантин при провале антивирусной проверки ClamAV
*   **Описание:** В случае обнаружения вредоносного ПО в ходе асинхронного сканирования ClamAV, версия файла немедленно изолируется (`FileVersion.status` = `INFECTED`)[cite: 6]. Так как при вызове `/complete` карточка `DocumentCard` переходит в статус `На проверке`, при фиксации заражения она принудительно возвращается в статус `Черновик` с флагом `quarantined = true`. Создание связей с бизнес-сущностями (`targetId` в смежных сервисах) полностью блокируется[cite: 6].
*   **Привязка к требованиям:** REQ-08-002[cite: 6].
*   **Критерий приемки (Вход в SPEC-11):** При получении статуса `INFECTED` сервис публикует событие `DocumentTechnicalCheckFailed`[cite: 6]. Карточка переходит обратно в `Черновик`, а любые попытки связать её с заявкой или экспертным делом возвращают ошибку `422 Unprocessable Entity`[cite: 6].

### BR-05: WORM-модель и иммутабельность версий
*   **Описание:** Реализуется модель хранения Write Once Read Many (Однократная запись — многократное чтение)[cite: 6]. Физическое удаление файлов или перезапись существующих S3-объектов при корректировках запрещены. Изменение документа всегда порождает новую запись в таблице `FileVersion` с инкрементом поля `versionNumber`[cite: 5, 6].
*   **Привязка к требованиям:** REQ-08-003, REQ-08-004[cite: 5, 6].
*   **Критерий приемки (Вход в SPEC-11):** Повторная загрузка файла в рамках существующего `documentId` создает новый UUID `fileVersionId`[cite: 6]. При этом старый файл в S3 остается доступным по своей исторической ссылке. `DocumentCard.actualFileVersionId` переключается на новую версию только после успешного завершения всех тех. проверок[cite: 2, 6].

### BR-06: Временное ограничение жизни ссылок (TTL Secure-Link)
*   **Описание:** Защищенная временная ссылка (`secureLinkRef`), генерируемая для скачивания или предпросмотра, формируется методом presigning с использованием протокола AWS Signature Version 4 (симметричное подписание на основе HMAC с использованием секретного ключа) и имеет строго ограниченное время жизни (Time-To-Live)[cite: 6]. По умолчанию TTL составляет 15 минут для внутренних пользователей и 30 минут для личного кабинета заявителя.
*   **Привязка к требованиям:** REQ-08-008[cite: 6].
*   **Критерий приемки (Вход в SPEC-11):** При попытке обращения к S3 по истечении установленного TTL (например, через 16 минут для внутренней ссылки) хранилище возвращает HTTP-статус `403 Forbidden`.

### BR-07: Публикация с валидацией профиля видимости (Visibility Profile Enforcement)
*   **Описание:** Документ не может быть переведен в статус `Направлен заявителю` и опубликован в ЛК, если его карточка содержит профиль видимости `INTERNAL`[cite: 6]. Публикация разрешена только для документов с профилями `APPLICANT_RESULT` или `APPLICANT_REQUEST_MATERIAL`[cite: 6].
*   **Привязка к требованиям:** REQ-08-010[cite: 6].
*   **Критерий приемки (Вход в SPEC-11):** Вызов метода `/publish` для документа со статусом видимости `INTERNAL` прерывается с ошибкой `409 Conflict` (бизнес-валидация прав публикации).

### BR-08: Фиксация состояний доставки (Delivery Receipt Lifecycle)
*   **Описание:** Каждое значимое действие заявителя с опубликованным документом в `cabinet-service` фиксируется созданием иммутабельной сущности `DeliveryReceipt` со следующими последовательными состояниями (`receiptType`)[cite: 6]:
    1.  `DELIVERED` — Документ отобразился в реестре личного кабинета (уведомление доставлено)[cite: 6].
    2.  `OPENED` — Пользователь открыл карточку документа или запустил встроенный просмотрщик[cite: 6].
    3.  `DOWNLOADED` — Пользователь инициировал скачивание бинарного файла на свое устройство[cite: 6].
*   **Привязка к требованиям:** REQ-08-009[cite: 6].
*   **Критерий приемки (Вход в SPEC-11):** Каждое состояние фиксируется отдельнойром в аудит-логе с временной меткой[cite: 6]. Переход в состояние `DOWNLOADED` невозможен без предварительной или одновременной регистрации состояния `DELIVERED`[cite: 6].

### BR-09: Изоляция ошибок пакетной загрузки (Batch Upload Atomicity)
*   **Описание:** При множественной или пакетной загрузке нескольких файлов в рамках одного бизнес-контекста, обработка каждой карточки документа происходит изолированно. Сбой технической валидации, обнаружение вируса или превышение лимита по одному из файлов не должны приводить к транзакционному откату (rollback) или блокировке загрузки остальных корректных файлов пакета.
*   **Привязка к требованиям:** REQ-08-001, REQ-08-004[cite: 6].
*   **Критерий приемки (Вход в SPEC-11):** Загрузка пакета из 3 файлов, где 1 файл заражен (статус `INFECTED`), оставляет две чистые карточки в валидном состоянии, а поврежденную переводит в `Черновик` с флагом `quarantined: true`.

### BR-10: Жесткий инвариант для SERVICE_RESOLUTION
*   **Описание:** Для документов типа «Заключение ведомства» (`documentType == SERVICE_RESOLUTION`) действует системный инвариант: профиль видимости обязан быть строго внутренним (`visibilityProfile == INTERNAL`). Любые попытки переопределить профиль на ЛК на этапе создания, редактирования или публикации пресекаются на уровне ядра доменной логики.
*   **Привязка к требованиям:** REQ-08-010[cite: 6].
*   **Критерий приемки (Вход в SPEC-11):** Попытка передать `visibilityProfile: APPLICANT_RESULT` для типа `SERVICE_RESOLUTION` на эндпоинты `/init` или `/publish` возвращает ошибку `409 Conflict`.

### BR-11: Доступность интерактивного предпросмотра (Preview Engine Restrictions)
*   **Описание:** Встроенный интерактивный предпросмотр содержимого документов в интерфейсах АРМ и ЛК разрешен только для файлов, чей MIME-тип входит в доверенный список отображаемых форматов (PDF, текстовые документы, стандартные растровые изображения). Для сложных, проприетарных или тяжелых инженерных форматов (например, CAD/DWG) автоматическая генерация слоев превью не выполняется.
*   **Привязка к требованиям:** REQ-08-008[cite: 6].
*   **Критерий приемки (Вход в SPEC-11):** При генерации ссылки для просмотра чертежа CAD система возвращает флаг `previewAvailable: false`, блокируя интерфейсный просмотрщик и предлагая только действие прямого скачивания.
