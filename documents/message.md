## 1. Точки интеграции со смежными сервисами

Сервис `document-service` (модуль загрузки документов и электронного документооборота) является центральным узлом документооборота в составе **Единой микросервисной системы**. Интеграционное взаимодействие осуществляется с **шестью** смежными сервисами. Модель обмена является смешанной: синхронные REST-вызовы применяются для команд и получения текущего состояния, а брокер Kafka — для доменных событий и обеспечения eventual consistency. Прямой доступ к базам данных чужих сервисов категорически запрещен.

| Смежный сервис

 | Назначение обмена

 | Инициатор взаимодействия

 | Протоколы и способ обмена

 |
| --- | --- | --- | --- |
| **`request-service`**<br> | Связь документов с заявкой, проверка комплектности, учет статусов и требований к комплекту документов.

 | **Обе стороны**: `document-service` валидирует контекст заявки; `request-service` запрашивает списки документов.

 | **REST**: `validateRequestContext`, `getRequiredDocuments`, `getDocumentsByRequest`.

<br>

## 2. Протоколы, обмен событиями и трассировка



* **REST API:** Применяется для выполнения синхронных команд, валидации бизнес-контекста, получения карточек документов и генерации временных ссылок. Владельцем контракта является сервис, предоставляющий API (для `document-service` — его собственная команда). Все изменяющие состояние команды обязаны поддерживать `idempotencyKey`.


* **Apache Kafka:** Используется для асинхронного уведомления смежных систем, разгрузки синхронного контура и обновления локальных проекций (eventual consistency). Гарантируется строгий порядок обработки сообщений внутри одного ключа (в качестве ключа используется `businessKey` для процессов или `documentId` для документов). Запрещено передавать binary content (тело файлов) через сообщения Kafka.


* **Единый Envelope события:** Все сообщения в Kafka публикуются строго через паттерн **Transactional Outbox** в едином формате конверта, содержащем метаданные трассировки: `eventId`, `eventType`, `eventVersion`, `occurredAt`, `producer`, `correlationId`, `causationId`, `businessKey` и изолированный `payload`.


* **Идемпотентность:** Consumer-ы событий обязаны проверять `eventId` на предмет дублирования и повторно не обрабатывать уже выполненные сообщения.

## 3. Внешние зависимости и инфраструктура



| Компонент | Выбранное технологическое решение | Целевой регламент и архитектурные правила

 |
| --- | --- | --- |
| **СУБД** | PostgreSQL

 | Прямой доступ к таблицам со стороны смежных сервисов **категорически запрещен**. Обмен только через контракты API.

 |
| **Брокер** | Apache Kafka

 | Схемы событий публикуются и версионируются в Schema Registry (форматы JSON Schema / Avro Schema). Обработка ошибок строится по схеме: 5 попыток (Consumer retry) с экспоненциальной задержкой $\rightarrow$ отправка в DLQ со всеми метаданными ошибки $\rightarrow$ ручной replay. Срок хранения сообщений в DLQ — не менее 14 календарных дней.

 |
| **Объектное хранилище** | S3-совместимое хранилище (MinIO)

 | Организуется версионирование (versioning), политики жизненного цикла (retention), изоляция по бакетам, карантин и расчет хэш-сумм (sha256).

 |

## 4. Безопасность и сквозная авторизация



* **Принцип эшелонированной защиты (Defense in Depth):** Проверка прав и авторизация запросов происходят как на уровне API Gateway, так и на уровне самого `document-service` (используется Spring Security в режиме OAuth2 Resource Server / JWT токенов).


* **Поля ограниченного доступа:** Все атрибуты, содержащие персональные данные, закрытые экспертные суждения или внутренние системные комментарии, явно маркируются в схемах данных мета-тегами `internalOnly` или `restricted`.


* **Изоляция сети:** Транспортное шифрование (mTLS) внутри контура Docker-сети является предметом отдельного развития, однако прикладная авторизация по JWT-токенам работает независимо от сетевого транспорта.

## 5. Обмен файлами документов



* **Передача по ссылкам:** Межсервисный обмен документами осуществляется **исключительно путем передачи метаданных и защищенных ссылок** на объектное хранилище (`objectStorageRef`, `presignedUrl`, `storageKeyHash`).


* **Ограничение на HTTP-тело:** Передача файла непосредственно в теле HTTP-запроса (через upload endpoint) происходит **только один раз** — в момент первоначальной загрузки файла клиентом (заявителем) или сервисом-источником в `document-service`. Дальнейшее движение файлов между микросервисами в виде байтовых потоков запрещено.

## 6. Политика отказоустойчивости и деградации (Circuit Breaker & Timeouts)



Проектом ответа жестко зафиксированы целевые параметры надежности для интеграционного слоя:

* **REST Timeouts:** Таймаут на установку соединения (`connectTimeout`) — **2 секунды**; таймаут на ожидание ответа (`responseTimeout`) — **5 секунд**. Для генерации защищенных ссылок в `cabinet-service` — до **3 секунд**. Тяжелые операции генерации и расчетов синхронно вызывать запрещено (только фон или асинхронные события).


* **Политика повторов (Retry REST):** Разрешена только для безопасных/идемпотентных операций (`GET`, `HEAD`) или `POST`/`PUT` запросов, содержащих `idempotencyKey`. Лимит — до 3 попыток с прогрессирующим backoff (200 мс $\rightarrow$ 1 сек $\rightarrow$ 3 сек). Неидемпотентные вызовы без ключа повторно автоматически не отправляются.


* **Circuit Breaker:** Активируется на синхронных вызовах ко всем 6 смежным сервисам. Размыкание цепи (переход в статус Open) происходит при уровне ошибок $\ge 50\%$ на скользящем окне из 20 вызовов (или при серии таймаутов). Полуоткрытое состояние (Half-open) проверяет контур 3 тестовыми запросами.


* **Поведение при недоступности смежных систем:**
* При отказе `request-service` загрузка возможна только в карантинный статус *«Черновик»* без связывания и без перехода на этап *«На проверке»*.


* Недоступность `expertise-service` или `review-service` полностью блокирует приемку и регистрацию экспертных карт и сравнительных протоколов.


* При отказе `workflow-service` запрещается изменять статус документа на *«На согласовании»* без валидного идентификатора маршрута (`routeId`).




* **Интерфейсный Fallback:** При авариях пользователю выводится дружелюбное функциональное сообщение без отображения технических стектрейсов. Все системные подробности логируются со сквозным `correlationId`.
