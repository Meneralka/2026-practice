# MZD-50: Реестр функциональных блоков document-service

**Модуль:** document-service (ТЗ-08, TZ-08-DOCUMENTS-EDO v1.0)
**Назначение артефакта:** свести де-факто сложившиеся пакеты в формальный перечень из 6 функциональных блоков — с составом, назначением и границами каждого.
**Источники:** SPEC-01 (границы, 11 зон), MZD-19 (классификация REQ), MZD-40 (каталог событий), SPEC-13 (enum), MZD-17/18 (IO и бизнес-правила).
**Статус:** проект для командного ревью.

## 2. Состав и границы по блокам

### 2.1. `document/` — ядро домена документа
**Назначение.** Владеет жизненным циклом карточки и файлов: единственный источник истины по `DocumentCard.status`.

**Состав:**
- `DocumentCard`: documentId, businessKey, documentType, status, visibilityProfile, реквизиты, ссылка на актуальную версию.
- `FileVersion`: загрузка init/complete, SHA-256, versionNumber, WORM (без перезаписи старых файлов), статус техпроверки.
- Технические проверки: MIME по содержимому (Apache Tika), лимит размера, антивирус (ClamAV), карантин.
- Объектное хранилище: S3-совместимое (RustFS/MinIO) через AWS SDK v2 — bucket/prefix policy, versioning, presigned URL, TTL.
- Статусная модель: тип-специфичные стейт-машины, матрица разрешённых переходов, `reasonCode`.

**Границы.** Никто, кроме этого блока, не пишет `status`. Комплектность заявки не считает (request-service), содержимое экспертных выводов не разбирает (expertise/review-service).

### 2.2. `link/` — связи с бизнес-объектами
**Назначение.** Учёт связей документа с внешними сущностями.

**Состав:** `DocumentLink` M:N; `linkType` ∈ {REQUEST, EXPERTISE_CASE, EXPERT_CARD, REVIEW, PRIMARY_EXPERTISE, REPORT, ROUTE} (SPEC-13 §2.2); идемпотентное создание, базовый CRUD.

**Границы.** Только факт связи и её тип. Вывод «комплект собран» делает request-service; данные соседних сервисов через REST/Kafka, без прямого доступа к их БД.

### 2.3. `registry/` — регистрация и нумерация
**Назначение.** Присвоение регистрационных номеров и ведение журналов.

**Состав:** регистрационные журналы и нумераторы; генерация уникального номера по `idempotencyKey` (Sequence, gaps allowed — MZD-23).

**Границы.** Номер выдаётся при переходе в `ACCEPTED`/`APPROVED`; неидемпотентная команда генерации не принимается.

### 2.4. `routing/` — маршруты согласования (ЭДО)
**Назначение.** Исполнение шагов согласования/утверждения в рамках домена.

**Состав:** `DocumentRoute`, `RouteTask`; изолированные стейт-машины маршрута; приём команд workflow-service.

**Границы.** Сквозной процесс не оркестрирует и SLA не считает — принимает команды маршрута только в рамках разрешённой матрицы переходов (владелец процесса — workflow-service).

### 2.5. `delivery/` — публикация и доставка заявителю
**Назначение.** Выдача документа во внешний контур (ЛК) и фиксация доставки.

**Состав:**
- Публикация по `visibilityProfile` (исполнение команды `requestPublication`).
- Защищённые временные ссылки (`secureLinkRef` → presigned URL по отдельному REST-вызову).
- Предпросмотр файлов в браузере.
- `DeliveryReceipt`: доставлен / открыт / скачан (MZD-34).

**Границы.** Не решает, *когда* публиковать (решение — report-service). `SERVICE_RESOLUTION` наружу не уходит никогда (8 слоёв защиты, SPEC-01 ч.3): для него `PUBLISHED` недостижим.

### 2.6. `integration/` — события и аудит
**Назначение.** Надёжная публикация доменных событий и сквозной аудит.

**Состав:**
- Transactional Outbox → топик `document.events.v1`.
- Конверт событий SPEC-14: `eventId`, `correlationId`, `causationId`, `businessKey`, `payload`.
- Реестр из 16 `DocumentEventType` (SPEC-13 §2.3); контрактно закреплены три из ТЗ: `DocumentUploaded`, `DocumentAccepted`, `DocumentPublished`.
- Аудит-журнал: кто, что, когда, с какого адреса, по какому основанию.

**Границы.** В Kafka — только метаданные и идентификаторы (`fileVersionId`, `secureLinkRef`, `objectStorageRef`); байты файлов и пути `s3://…` в payload запрещены. Поля `internalOnly`/`restricted` не покидают внутренний контур.

## 3. Карта покрытия зон SPEC-01

Все 11 зон ответственности SPEC-01 распределены между 6 блоками без пробелов и пересечений:

| Зона SPEC-01 | Блок |
|--------------|------|
| 1. Карточка документа | `document/` |
| 2. Файлы и версии | `document/` |
| 3. Технические проверки | `document/` |
| 4. Объектное хранилище | `document/` |
| 5. Связи | `link/` |
| 6. Нумерация | `registry/` |
| 7. ЭДО (маршрут) | `routing/` |
| 8. Статусная модель | `document/` |
| 9. Публикация и доставка | `delivery/` |
| 10. События | `integration/` |
| 11. Аудит | `integration/` |

**Открытый вопрос ревью.** Аудит (зона 11) отнесён к `integration/` как сквозная функция, питающаяся от Outbox-потока. Если аудит вырастет до самостоятельного хранилища/API, его следует выделить в отдельный блок — тогда перечень станет из 7 блоков.
