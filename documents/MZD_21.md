# MZD-21: Входные/Выходные интерфейсы (I/O) и каталог событий интеграционного контура

## 1. Сущность `DocumentLink` (Связи объектов)
Связи приведены в строгое соответствие со спецификацией **SPEC-13** (разрешение конфликта MZD-5 vs MZD-6):
* **Кардинальность:** `Many-to-Many` (M:N). Один документ может быть привязан к нескольким бизнес-объектам, и один объект содержит множество документов.
* **Название поля типа связи:** `linkType` (вместо устаревшего `relatedEntityType`).
* **Допустимые значения связей (7 типов, SPEC-13 §2.2: REQ-08-005 + PRIMARY_EXPERTISE):**
  1. `REQUEST` — Заявка
  2. `EXPERTISE_CASE` — Дело
  3. `EXPERT_CARD` — Экспертная карта
  4. `REVIEW` — Проверка
  5. `PRIMARY_EXPERTISE` — Первичная экспертиза (необходима для кросс-документальной валидации)
  6. `REPORT` — Отчет
  7. `ROUTE` — Маршрут

## 2. Интерфейсы пакетов `registry/` и `routing/`

### Регистрационные журналы (`registry/`)
* **Вход:** `GenerateNumberRequest` (содержит `documentType`, `journalId`, `idempotencyKey`). Параметр «код подразделения» исключен как не имеющий источника в ТЗ и вынесен в SPEC-16.
* **Выход:** `RegistrationNumberResponse` (строковый сформированный номер, ID записи журнала).
* **Идемпотентность:** Гарантируется сервером. Повторный запрос с идентичным `idempotencyKey` в рамках одной пары (`documentType`, `journalId`) возвращает сохраненный номер из кэша/БД без повторного инкремента.

### Маршрутизация (`routing/`)
* **Вход:** `ExecuteTaskCommand` (содержит `idempotencyKey`, `taskId`, `action`).
* **Идемпотентность:** Реализована на уровне бизнес-логики сервиса через таблицу обработанных ключей идемпотентности по **ФТ-35**. Обработанный `idempotencyKey` сохраняется в БД; при дублировании запроса повторная смена статуса блокируется, возвращается текущее актуальное состояние. Поле `@Version` используется исключительно на уровне JPA как оптимистическая блокировка от конкурентных гонок данных, но не заменяет функциональную идемпотентность.

## 3. Каталог событий Outbox (Синхронизировано со SPEC-14)

Список событий не является исчерпывающим и расширяется до 16 системных событий (согласно общему реестру). Все сообщения оборачиваются в единый конверт (Envelope) строго по **SPEC-14**:

```json
{
  "eventId": "UUID",
  "eventVersion": "1.0.0",
  "eventType": "DocumentUploaded", 
  "occurredAt": "2026-07-12T10:15:00Z",
  "producer": "document-service",
  "causationId": "UUID-родительского-события-команды",
  "correlationId": "UUID-сквозного-процесса",
  "businessKey": "ID-ключ-карточки-документа",
  "payload": {
    "fileId": "UUID",
    "mimeType": "application/pdf",
    "fileVersionId": "UUID"
  }
}
