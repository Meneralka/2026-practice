# MZD-35: Автоматические проверки файлов на техническом конвейере

Настоящая спецификация регламентирует состав, условия, логику выполнения и последствия автоматических проверок файлов при загрузке. Виды файловых проверок соответствуют enum `CheckType` (SPEC-13 §3.2), причины отказов — `ReasonCode` (SPEC-13 §3.3), статусы версии — MZD-16 / MZD-33.

## 1. Реестр проверок конвейера

Для каждого загружаемого файла выполняется 6 типов проверок:

1. **Техническая валидация контента через Apache Tika** (`CheckType.MIME_TYPE`)
2. **Проверка жёсткого лимита размера** (`CheckType.FILE_SIZE`)
3. **Контроль целостности по контрольной сумме SHA-256** (`CheckType.INTEGRITY`)
4. **Антивирусное сканирование ClamAV** (`CheckType.ANTIVIRUS`)
5. **Проверка прав доступа (ACL / RBAC)**
6. **Валидация профиля видимости (Visibility Profile Enforcement)**

## 2. Детальное описание проверок

### 2.1. Техническая валидация контента (Apache Tika)
* **Момент вызова:** асинхронно, после перехода версии в `CHECKING` (карточка — в `UNDER_REVIEW`, MZD-32 §3.2).
* **Что происходит:** анализатор считывает первые байты файла (`magic bytes`) для определения истинного MIME-типа; результат сравнивается с разрешённой матрицей MIME-типов для указанного `documentType`.
* **Действие при отказе:** версия переводится в `INVALID` (`reasonCode = MIME_TYPE_NOT_ALLOWED`), карточка возвращается в `DRAFT` с флагом `quarantined: true`, физический файл удаляется из карантина, публикуется `DocumentTechnicalCheckFailed`.

### 2.2. Проверка жёсткого лимита размера (`FILE_SIZE`)
* **Момент вызова:** трижды — превентивно при `/init`, физически на уровне S3 через `content-length-range` в POST Policy, и финально синхронно при `/complete` посредством HEAD-запроса к бакету.
* **Что происходит:** сравнение фактического размера объекта в байтах с максимально допустимым значением конфигурационной матрицы для данного `documentType`.
* **Действие при отказе:** запрос `/init` или `/complete` завершается ошибкой `413 Payload Too Large`, сессия загрузки аннулируется. На этапе `/complete` версия помечается `INVALID` (`reasonCode = FILE_SIZE_EXCEEDED`).

### 2.3. Контроль целостности (`INTEGRITY`, SHA-256)
* **Момент вызова:** синхронно на этапе `/complete`.
* **Что происходит:** клиент обязан передать SHA-256 загружаемого файла; `document-service` вычисляет хэш объекта в S3-карантине и сопоставляет с предоставленным.
* **Действие при отказе:** транзакция прерывается, `/complete` возвращает `422 Unprocessable Entity`, версия переводится в `INVALID` (`reasonCode = HASH_MISMATCH`).

### 2.4. Антивирусное сканирование (`ANTIVIRUS`, ClamAV)
* **Момент вызова:** асинхронно, в рамках конвейера проверок ИБ (после HEAD-валидации размера и контрольной суммы).
* **Что происходит:** служба ClamAV сканирует файл, временно размещённый в `quarantine-bucket`.
* **Действие при отказе:** версия немедленно помечается `INFECTED` (`reasonCode = ANTIVIRUS_INFECTED`), бинарное тело уничтожается в карантине, карточка возвращается в `DRAFT` с флагом `quarantined = true`, публикуется `DocumentTechnicalCheckFailed`.

### 2.5. Проверка прав доступа (ACL / RBAC)
* **Момент вызова:** синхронно при `/init`, `/complete`, `/publish` и при запросах на чтение/скачивание.
* **Что происходит:** проверка полномочий вызывающего субъекта (пользователь ЛК, сотрудник ведомства, системная учётная запись) на операцию над данной категорией документов.
* **Действие при отказе:** запрос отклоняется с `403 Forbidden`; изменений в БД и S3 не производится.

### 2.6. Валидация профиля видимости (Visibility Profile Enforcement)
* **Момент вызова:** синхронно на этапе `/init` и при вызове `/publish`.
* **Что происходит:** проверка соответствия `visibilityProfile` типу документа. Жёсткий инвариант (SPEC-13 §2.1): `documentType == SERVICE_RESOLUTION` ⇒ `visibilityProfile == INTERNAL`. Публикация во внешний контур ЛК разрешена только для `APPLICANT_RESULT` и `APPLICANT_REQUEST_MATERIAL`.
* **Действие при отказе:** запрос отклоняется с `422 Unprocessable Entity` (недопустимый профиль — нарушение матрицы, конвенция кодов MZD-25/MZD-32 §5).

## 3. Политика отказоустойчивости и недоступности сервисов проверки

Ни одна из автоматических проверок безопасности не может быть пропущена или проигнорирована по причине технического сбоя проверяющей инфраструктуры.

* **Принцип закрытости по умолчанию (Fail-Secure):** при падении, перегрузке или недоступности любого сервиса проверки (ClamAV, Apache Tika, база конфигураций прав) система блокирует дальнейшее продвижение файла по конвейеру.
* **Поведение при недоступности ClamAV / Tika:**
  * Файл остаётся в статусе `CHECKING` (версия), карточка — в `UNDER_REVIEW`.
  * Запускается автоматическая политика повторных попыток (Exponential Backoff, 5 попыток в течение 10 минут).
  * Если по истечении лимита связь не восстановилась, в систему мониторинга (Sentry/Opsgenie) отправляется критический алерт; кейс передаётся на ручной разбор ИБ. При подтверждённой невозможности выполнить проверку версия переводится в `INVALID` с `reasonCode = ANTIVIRUS_SCAN_FAILED`.
  * Любые запросы бизнес-логики на чтение, связывание, подписание или публикацию такого документа отклоняются с `409 Conflict` (файл не в `CLEAN` — конвенция ФТ-07, MZD-33 §5, MZD-45 E4).
