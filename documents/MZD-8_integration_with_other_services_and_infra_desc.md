## 1. Точки интеграции с соседними сервисами

Сервис (`document-service`) интегрирован с **шестью** соседними сервисами. По ТЗ (раздел 8) все интеграции **двунаправленные** (`вход/выход`): сервис как получает, так и передаёт данные, статусы, задачи, события и документы. Модель «один вход — один выход» неприменима; сервис является центральным узлом документооборота в контуре.

| Соседний сервис | Направление | Предмет обмена |
|---|---|---|
| `request-service` | вход/выход | данные, статусы, задачи, события, документы |
| `expertise-service` | вход/выход | данные, статусы, задачи, события, документы |
| `review-service` | вход/выход | данные, статусы, задачи, события, документы |
| `report-service` | вход/выход | данные, статусы, задачи, события, документы |
| `cabinet-service` | вход/выход | данные, статусы, задачи, события, документы |
| `workflow-service` | вход/выход | данные, статусы, задачи, события, документы |

**Доменные события**, публикуемые сервисом в брокер: `DocumentUploaded`, `DocumentAccepted`, `DocumentPublished` — с версионируемой схемой и `correlationId` для сквозной трассировки.

**Объём интеграционной работы:** шесть двунаправленных контрактов + публикация доменных событий. Это существенно больше, чем линейная цепочка, и должно закладываться в оценку трудозатрат backend-разработки соответственно.

**Открытые вопросы:**
- Не зафиксированы контракты API (схемы, версионирование) по каждому из шести направлений.
- Не определена стратегия деградации при недоступности соседнего сервиса (таймауты, ретраи, circuit breaker).

## 2. Доступные протоколы и обмен событиями

- **Синхронное взаимодействие:** HTTP во внутренней изолированной сети (Docker-сеть). При соединении с внешними источниками — обязательно HTTPS.
- **Асинхронное взаимодействие:** **Apache Kafka** (Spring for Apache Kafka) — принятое решение. Доменные события публикуются в брокер по паттерну **outbox**, схема сообщений версионируется, каждое событие несёт `correlationId`.

Конкретная разбивка «что идёт синхронно по HTTP, а что асинхронно через Kafka» по каждому из шести направлений определяется на уровне реализации; доменные события (`DocumentUploaded/Accepted/Published`) идут через Kafka.

**Ограничение / осознанный компромисс:** трафик внутри Docker-сети передаётся без транспортного шифрования (доверие к изоляции контура). Приемлемо для изолированной сети; авторизация запросов при этом обеспечивается на прикладном уровне (см. §4).

## 3. Внешние зависимости сервиса

| Зависимость | Решение | Статус |
|---|---|---|
| СУБД | PostgreSQL | Определено |
| Брокер сообщений | Apache Kafka + Spring for Apache Kafka | Определено |
| Объектное хранилище | MinIO (доступ по S3 API через AWS SDK for Java v2) | Определено |
| Кэш | Не используется | Определено |

Объектное хранилище — **MinIO**, доступ осуществляется по **протоколу S3** через AWS SDK for Java v2. S3 здесь — это протокол доступа к хранилищу, на который завязаны presigned URLs (REQ-009), versioning (REQ-004) и object lock. S3 не является отдельной внешней зависимостью и не исключён — это API-контракт работы с MinIO.

## 4. Безопасность и аутентификация

Межсервисная авторизация **присутствует и является требованием ТЗ** (раздел 10): межсервисные вызовы выполняются по защищённому каналу с проверкой токенов; доступ к API проверяется на уровне gateway и на уровне самого сервиса.

- **Авторизация запросов:** Spring Security в режиме **OAuth2 Resource Server (JWT)**. Каждый входящий запрос проверяется по JWT — как на gateway, так и в самом сервисе (defense in depth).
- **Транспорт внутри контура:** допускается без TLS — осознанный компромисс, обоснованный сетевой изоляцией Docker-сети. Прикладная авторизация по JWT при этом действует независимо от транспорта.
- **Внешние соединения:** HTTPS обязателен.

**Уточнение к модели угроз:** формулировка «доверяем только сети» некорректна — сетевая изоляция не является единственным барьером. Барьер прикладного уровня (JWT-проверка) присутствует. Отсутствует только межсервисный mTLS, что для изолированного контура допустимо и может быть добавлено позже.

## 5. Обмен документами

- Документы передаются **ссылками на объектное хранилище (presigned URLs)**, а не телом HTTP-запроса. Это снимает ограничения на размер тела и снижает нагрузку на межсервисную сеть.
- Хранилище — MinIO по S3 API; заложены **presigned URLs** (REQ-009), **versioning** (REQ-004) и **object lock**. Настройка и эксплуатация этих механизмов — зона ответственности DevOps.

## 7. Наблюдаемость

Наблюдаемость покрыта по всем трём столпам — это выбранные решения стека, а не незакрытые вопросы. Открытым остаётся только уровень реализации/настройки.

| Столп | Решение стека | Статус |
|---|---|---|
| Логи | Loki / ELK, централизованный сбор | Выбрано |
| Метрики | Micrometer + Prometheus | Выбрано |
| Трассировка | OpenTelemetry / Jaeger | Выбрано |
| Health / Readiness | Spring Boot Actuator | Выбрано |
| Корреляция | `traceId` / `correlationId` (сквозные, НФТ по ТЗ разд. 11) | Требование ТЗ |

Actuator health/readiness и сквозной `traceId`/`correlationId` — отдельное нефункциональное требование ТЗ (раздел 11) и одна из первых задач DevOps. Остаётся проработать реализацию (инструментирование, экспорт метрик, дашборды, sampling трассировки), а не выбор инструментов.
