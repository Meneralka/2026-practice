# MZD-46: Тесты требований безопасности

**Область:** два инварианта — (1) служебная резолюция не покидает внутренний контур; (2) файлы не проходят через брокер.
**Основания:** MZD-22 §2.5 (жёсткий guard), стейкхолдер-док §2 и §6.1 (запрет binary в Kafka), ТЗ р.10 (RBAC/ABAC), SPEC-04 (`visibilityProfile`).
**Пересечение с блоком E (MZD-45):** транзит `SERVICE_RESOLUTION → PUBLISHED` покрыт тестом **E1.8** — здесь не дублируется. Блок F закрывает утечку **в обход статуса**.

## F1. Служебная резолюция не публикуется и не доходит до заявителя

Защита эшелонированная: даже если один барьер пробит, остальные должны удержать. Поэтому каждый барьер — отдельный тест.

| # | Барьер | Сценарий | Ожидание | Слой |
|---|---|---|---|---|
| F1.1 | Транзит статуса | `SERVICE_RESOLUTION` → `PUBLISHED` | **403**, `SecurityAccessDeniedException` | см. E1.8 |
| F1.2 | Эндпоинт публикации | `POST /documents/{id}/publish` для `SERVICE_RESOLUTION` | **403**, статус не изменился, событие не опубликовано | SCC + INT |
| F1.3 | Выдача в ЛК | `getPublishedDocuments(applicantId, businessKey)` при наличии резолюций в контуре | в ответе **нет** ни одной `SERVICE_RESOLUTION` | SCC + INT |
| F1.4 | Secure-link | `POST /documents/{id}/secure-links` для `SERVICE_RESOLUTION` | **403**, ссылка не создана | SCC |
| F1.5 | Прямое чтение карточки | `GET /documents/{id}` (`SERVICE_RESOLUTION`) под ролью заявителя | **403** (ABAC, ТЗ р.10) | SCC + INT |
| F1.6 | Список с фильтром видимости | `GET /documents?visibilityProfile=APPLICANT_RESULT` / `APPLICANT_REQUEST_MATERIAL` | `SERVICE_RESOLUTION` отсутствует в выборке | INT |
| F1.7 | Delivery receipt | `POST /documents/{id}/delivery-receipts` для `SERVICE_RESOLUTION` | **409** (документ не в `PUBLISHED`) | SCC |
| F1.8 | Событие публикации | прогон полного цикла резолюции `DRAFT → UNDER_APPROVAL → APPROVED → ARCHIVED` | в `document.events.v1` **ни одного** `DocumentPublished` с `documentType=SERVICE_RESOLUTION` | INT |
| F1.9 | Утечка метаданных | любой ответ cabinet-контура (F1.3/F1.6) | отсутствуют не только файлы, но и **метаданные**: `title`, `fileName`, `documentId` резолюции | INT |

**F1.9 — ключевой.** Утечка факта существования документа («у вас есть служебная резолюция №…») — уже нарушение, даже без доступа к файлу.

## F2. Файлы не уходят через брокер сообщений

**Инвариант (стейкхолдер-док §2, §6.1; закреплён в MZD-40 §2):** в Kafka — только метаданные и идентификаторы. Binary content запрещён.

| # | Сценарий | Ожидание | Слой |
|---|---|---|---|
| F2.1 | Схема любого события из `document.events.v1` | в JSON Schema/Avro **нет** полей типа `bytes` / `base64` / `content` / `fileData` | SCC (producer verifier) |
| F2.2 | Прогон полного цикла (upload → accept → publish) | все события несут только идентификаторы (`fileVersionId`, `secureLinkRef`-идентификатор) и `sha256`; ни тела файла, ни presigned URL, ни путей хранилища (`s3://…`) | INT |
| F2.3 | Попытка опубликовать событие с binary-полем | отклонено на этапе валидации схемы (Schema Registry / producer verifier), в топик не попадает | INT |
| F2.4 | Размер сообщения | ни одно событие не превышает разумный порог метаданных (не растёт пропорционально размеру файла: 1 КБ vs 50 МБ файл → размер события не меняется) | INT |
| F2.5 | Межсервисный REST | `registerReportDocument` / `registerExpertCardDocument` / `registerReviewProtocol` с телом файла вместо `fileDescriptor` → **415** (тело файла принимает только upload-endpoint) | SCC |
| F2.6 | Единственное исключение (позитив) | `POST /documents/uploads/{uploadId}/complete` — загрузка в `document-service`: тело файла принимается **только здесь**; для остальных endpoint'ов — отказ | SCC |

**F2.4 — самый показательный тест:** он ловит утечку, которую схема может пропустить (например, если файл прокрался в `payload` как строка). Загружаем 1 КБ и 50 МБ, сравниваем размер события — он обязан быть одинаковым.

## F3. Смежные проверки безопасности (ТЗ р.10)

| # | Сценарий | Ожидание | Слой |
|---|---|---|---|
| F3.1 | Запрос без JWT | **401** на gateway и на сервисе (defense in depth) | SCC |
| F3.2 | Документ чужой организации | **403** (ABAC) | SCC + INT |
| F3.3 | Чужой документ в своей организации | **403** (RBAC по `accessScope`) | SCC + INT |
| F3.4 | JWT валиден на gateway, но сервис вызван напрямую в обход gateway | сервис самостоятельно проверяет JWT → **401/403**, а не пропускает | INT |

**F3.4** проверяет, что «доверяем сети» не превратилось в дыру: изоляция Docker-сети не должна быть единственным барьером.

## Решения по бывшим открытым вопросам блока F

1. **`secureLinkRef` в payload `DocumentPublished`** — закрыто в MZD-40 §2: это **идентификатор** защищённой ссылки, а не presigned URL. Временный URL получатель запрашивает отдельным REST-вызовом. Тест F2.2 проверяет отсутствие сырых URL в событиях.
2. **`objectStorageRef` в событиях** — закрыто: пути хранилища (`s3://…`) в payload не включаются, вместо них `fileVersionId` (MZD-40 §2). Тест F2.2 проверяет и это.
3. **Retention DLQ ≥ 14 дней** — снят вслед за п.1: в DLQ оседают только метаданные и идентификаторы (MZD-42 §3), рабочих ссылок на файлы там нет.
