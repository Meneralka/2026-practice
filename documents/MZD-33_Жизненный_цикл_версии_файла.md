# MZD-33: Жизненный цикл версии файла и правила карантина

Настоящая спецификация определяет внутренний жизненный цикл версий файлов (`FileVersion`), правила управления карантином, механизм переключения актуальной версии в карточке документа (`DocumentCard`) и жёсткие блокировки переходов бизнес-логики до завершения проверок ИБ. Статусная модель версии — по MZD-16 / SPEC-02.

## 1. Реестр статусов версии файла (`FileVersion.status`, MZD-16 §1)

1. **`NEW`**: Запись создана в рамках первой фазы двухфазной загрузки (`/uploads/init`), выделены метаданные и сгенерирована presigned-политика; бинарный контент в S3 ещё не передан.
2. **`UPLOADED`**: Файл успешно загружен клиентом в карантинный бакет S3, клиент вызвал `/uploads/complete`, синхронные проверки (HEAD-размер, SHA-256) пройдены; управление возвращено в `document-service`.
3. **`CHECKING`**: Файл передан на асинхронный конвейер технической валидации (Apache Tika) и антивирусного сканирования (ClamAV).
4. **`CLEAN`**: Успешный финальный статус. ClamAV подтвердил отсутствие вирусов, Apache Tika — соответствие MIME-типа вайтлисту. Файл физически перемещён в основное хранилище `main-bucket`.
5. **`INFECTED`**: Терминальный статус провала проверки ИБ. ClamAV обнаружил вредоносный код. Файл подлежит немедленной изоляции и удалению.
6. **`INVALID`**: Терминальный статус провала технической валидации: расхождение реального MIME-типа с заявленным расширением, несовпадение SHA-256 либо превышение лимита размера. Причина фиксируется в `reasonCode` и детализируется в `checkDescription` (MZD-16 §2.1).

**Отмена/просрочка загрузки** отдельным статусом не моделируется: если пользователь прервал загрузку или истёк 2-часовой TTL сессии до вызова `/complete`, сессия загрузки аннулируется, версия остаётся в `NEW`, не участвует в бизнес-логике и вычищается воркером (см. §4).

## 2. Линейный жизненный цикл версии файла

### `NEW` (Создана)
* **Триггер:** успешный вызов `/uploads/init`.
* **Действие системы:** резервирование UUID версии в БД, выдача PUT/POST-политики для S3-карантина.

### `UPLOADED` (Загружена)
* **Триггер:** успешный вызов `/uploads/complete`.
* **Действие системы:** HEAD-запрос к S3 (размер), сверка SHA-256, блокировка возможности скачивания данной версии, отправка ID версии в очередь конвейера асинхронных проверок.

### `CHECKING` (На проверке)
* **Триггер:** конвейер взял версию в обработку.
* **Действие системы:** карточка документа переводится в `UNDER_REVIEW` (кроме `SERVICE_RESOLUTION`, см. MZD-32 §3.2); запуск Tika и ClamAV.

### `CLEAN` (Чистая) — *успешный финал*
* **Триггер:** получение успешных результатов от ClamAV и Apache Tika.
* **Действие системы:** копирование бинарного объекта из `quarantine-bucket` в `main-bucket` (Object Lock), удаление временного объекта из карантина, обновление поля `actualFileVersionId` в родительской карточке, публикация `DocumentTechnicalCheckPassed`.

### `INFECTED` (Заражённая) — *критический отказ*
* **Триггер:** ClamAV возвращает положительный результат на сигнатуру вируса.
* **Действие системы:** немедленное физическое удаление файла из `quarantine-bucket`, установка флага `quarantined: true`, публикация `DocumentTechnicalCheckFailed` (`reasonCode = ANTIVIRUS_INFECTED`).

### `INVALID` (Невалидная) — *технический отказ*
* **Триггер:** Tika выявляет подмену расширения/MIME-типа (`MIME_TYPE_NOT_ALLOWED`), HEAD-запрос — превышение размера (`FILE_SIZE_EXCEEDED`), либо не совпал SHA-256 (`HASH_MISMATCH`).
* **Действие системы:** остановка обработки, удаление файла из бакета карантина, публикация `DocumentTechnicalCheckFailed`.

## 3. Правило актуальной версии (`actualFileVersionId`)

Система реализует иммутабельную WORM-модель версионирования (MZD-16 §2.2):

1. **Единственность активной версии:** за хранение указателя на актуальный файл отвечает поле `DocumentCard.actualFileVersionId`. Флага `isCurrent` на `FileVersion` нет — иначе два источника истины.
2. **Атомарность переключения:** `actualFileVersionId` перезаписывается UUID новой версии **строго** после перехода новой версии в `CLEAN`.
3. **Поведение при ошибках:** если новая версия перешла в `INFECTED` или `INVALID`, `actualFileVersionId` **не меняется** — продолжает указывать на предыдущую чистую версию (или остаётся `null`, если чистых версий не было). Признак карантина для UI вычисляется как `actualFileVersionId == null && последняя_версия.status IN (INFECTED, INVALID)`.
4. **История версий:** старые версии сохраняют статус `CLEAN` и доступны в истории карточки для аудита. Удаление исторических версий из `main-bucket` запрещено.

## 4. Карантин и правила изоляции файлов

Бакет карантина (`quarantine-bucket`) — изолированная зона, полностью закрытая для прямого доступа пользователей и смежных микросервисов.

* **Изоляция доступа:** к бакету карантина имеют доступ только внутренний системный аккаунт `document-service` (запись и удаление) и агент сканирования ClamAV (только чтение).
* **Запрет генерации ссылок:** для версий в статусах `NEW`, `UPLOADED`, `CHECKING`, `INFECTED`, `INVALID` генерация временных ссылок (`secureLinkRef`) полностью заблокирована на уровне API. Ссылки выдаются только на версии `CLEAN` (MZD-36).
* **Очистка карантина:** воркер очистки раз в сутки удаляет из `quarantine-bucket` объекты старше 24 часов, для которых не был вызван `/complete` (зависшие или брошенные загрузки), и абортирует незавершённые S3-multipart-загрузки.

## 5. Жёсткое правило блокировки движения документа

> **ГЛАВНЫЙ СИСТЕМНЫЙ ИНВАРИАНТ (ФТ-07):**
> Карточка документа не имеет права изменять свой статус по бизнес-цепочке, пока её текущая (загружаемая) версия файла не перейдёт в статус `CLEAN`.

### Механизм блокировки:
1. Пока версия находится в `NEW`, `UPLOADED` или `CHECKING`, любые попытки вызвать бизнес-методы (публикация `/publish`, привязка через `targetId`, согласование, приёмка) блокируются бизнес-валидатором. API возвращает `409 Conflict`:
   > **Действие заблокировано**
   > Документ находится в процессе технической проверки безопасности. Пожалуйста, дождитесь завершения сканирования файла.
2. Если версия признана `INFECTED` или `INVALID`, карточка возвращается в `DRAFT` с флагом `quarantined: true`. Из этого состояния допустима только загрузка новой версии файла.
3. Движение карточки по жизненному циклу разрешается **только после** перехода версии в `CLEAN`; негативные тест-кейсы — MZD-45, блок E4.
