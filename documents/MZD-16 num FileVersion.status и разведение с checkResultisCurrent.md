# MZD-16 / SPEC-02: Разграничение статусных моделей сущностей DocumentCard и FileVersion

## 1. Закрытый перечень значений FileVersion.status

Сущность `FileVersion` имеет собственную независимую статусную модель, отражающую исключительно физическое состояние конкретного файла и результаты его прохождения через технические конвейеры системы[cite: 5, 6]. 

| Значение `FileVersion.status` | Описание |
| :--- | :--- |
| **`NEW`** | Запись создана в рамках первой фазы двухфазной загрузки (`uploads/init`), физический файл в объектное хранилище еще не отправлен. |
| **`UPLOADED`** | Файл успешно загружен клиентом в S3, метаданные проверены, управление возвращено в `document-service` (`uploads/complete`)[cite: 5, 6]. |
| **`CHECKING`** | Файл передан на конвейер технической валидации и антивирусного сканирования ClamAV[cite: 5, 6]. |
| **`INFECTED`** | Обнаружена угроза безопасности или вредоносный код по результатам ClamAV. Файл изолируется[cite: 6]. |
| **`INVALID`** | Файл не прошел проверку структуры или формата силами Apache Tika (битый файл, подмена расширения MimeType и т.д.)[cite: 6]. |
| **`CLEAN`** | Технический аудит и антивирусная проверка успешно пройдены[cite: 2, 6]. Файл готов к использованию в бизнес-логике. |

## 2. Разведение семантики полей: status, checkDescription и актуальность версий

Для устранения архитектурного долга и перекрытия семантики назначение полей распределено следующим образом[cite: 2]:

### 2.1. `FileVersion.status` vs `FileVersion.checkDescription`
*   **`status` (Enum):** Отражает текущую *стадию конвейера обработки* или технический статус файла. Служит для быстрой фильтрации, индексации и роутинга событий.
*   **`checkDescription` (Text / JSON):** **Не дублирует статус.** Хранит *детализированный текстовый отчет или лог проверок* для системных администраторов и координаторов[cite: 2, 6]. 
    * *Пример для статуса `INVALID`:* `{"tikaResult": "Failed", "expected": "application/pdf", "actual": "application/x-executable", "reason": "MimeType spoofing detected"}`.
    * *Пример для статуса `INFECTED`:* `{"virusScan": "Malicious", "threatName": "Eicar-Test-Signature"}`.

### 2.2. Идентификация актуальной версии и признака Карантина
Поле `isCurrent` (булево flag-поле на уровне версии) полностью удалено из сущности `FileVersion` для исключения Race Condition при WORM-модели[cite: 2, 6].
*   **Правило определения актуального файла:** Единственным источником истины для получения текущей версии является поле **`DocumentCard.actualFileVersionId`** (ссылка на UUID последней валидной версии файла)[cite: 2, 6].
*   **Явный признак Карантина (Выход для UI @NITF1S):** Для разграничения состояний «только загрузили» и «заражен / заблокирован конвейером» вводится явное вычисляемое состояние (или флаг на уровне ответа API). 
    * Если файл провалил техпроверку (`INFECTED` или `INVALID`), поле **`DocumentCard.actualFileVersionId` остается равным `null`** (или сохраняет значение предыдущей валидной версии)[cite: 2, 6], а в историю версий пишется `FileVersion.status` с соответствующей ошибкой[cite: 6].
    * На уровне API для фронтенда отдается явное свойство структуры: `quarantined: true` (вычисляется как `actualFileVersionId == null && последняя_версия.status IN (INFECTED, INVALID)`). Это позволяет UI корректно подсветить ошибку и вывести причину из `checkDescription`.

## 3. Матрица пересечения с DocumentCard.status (Согласовано с @ArtemKvvs)

### Архитектурная оговорка по стейт-машинам
Технический конвейер проверок файлов `document-service` **не переводит автоматически карточку в целевые бизнес-статусы** (такие как `Принят`, `Утвержден` и т.д.)[cite: 6]. Смена бизнес-статуса карточки происходит строго в рамках индивидуальной стейт-машины конкретного `documentType` и инициируется исключительно внешними сервисами-владельцами процессов (Бизнес-контекст)[cite: 6].

| Исходный статус `DocumentCard` | `FileVersion.status` | Результирующий статус `DocumentCard` | Системное действие / Инициатор / Ограничение стейт-машины |
| :--- | :--- | :--- | :--- |
| **Черновик**[cite: 6] | `NEW` $\rightarrow$ `UPLOADED`[cite: 6] | **Черновик**[cite: 6] | Ожидание завершения физической загрузки. Допустимо для всех типов документов[cite: 6]. |
| **Черновик** | `CHECKING` | **Черновик** | Автопроверки живут внутри `Черновика` (Q-1, close-questions): технический результат несёт `FileVersion.status`, статус карточки не дублирует его. |
| **Черновик** | `INFECTED` \| `INVALID` | **Черновик** (Карантин) | Карточка остаётся в `Черновике`, блокируется от бизнес-связей, выставляется флаг `quarantined: true`. Требуется повторная загрузка файла. |
| **Черновик** | `CLEAN` | **На проверке** | Автоматический переход `DRAFT` $\rightarrow$ `UNDER_REVIEW` при `CLEAN` + валидном контексте бизнес-объекта (Q-1). *Ограничение:* для `SERVICE_RESOLUTION` переход не выполняется — `UNDER_REVIEW` исключён её графом, карточка остаётся в `Черновике` до запуска согласования. Дополнительно публикуется доменное событие `TechnicalCheckPassed`. |
| *Любой валидный статус* | `CLEAN`[cite: 6] | **Принят** (Для `REQUEST_MATERIAL` / `EXPERT_CARD` / `COMPARATIVE_PROTOCOL`)[cite: 6] | Перевод осуществляется **исключительно внешним бизнес-сервисом** (`request-service`, `expertise-service` или `review-service`) после анализа факта успешной проверки файла[cite: 6]. *Для `EXPERT_CONCLUSION` статус `Принят` запрещен стейт-машиной.* |
| **Требуется замена** | Новая `FileVersion` достигла `CLEAN` | **На проверке** | Автоматический возврат карточки на ручную проверку после успешных автопроверок новой версии файла (MZD-22 §2.1; если применимо по типу документа). Пока новая версия в `NEW`/`CHECKING` или провалила проверки — карточка остаётся в `Требуется замена`. |
| **Архив**[cite: 6] | Любой | **Архив** (Блокировка)[cite: 6] | Любые операции добавления или изменения версий `FileVersion` для карточки в статусе `Архив` жестко запрещены на уровне API[cite: 6]. |
