# MZD-61 / SPEC-12: Сервисы обработки данных

Настоящая спецификация сводит ключевую бизнес-логику обработки документов в структурированный каталог сервисов. Спецификация описывает пайплайны валидации, работу с S3-хранилищем, публикацию документов и регистрацию квитанций.

---

## 1. Пайплайн валидации (`DocumentValidationPipeline`)

Сервис обеспечивает последовательное техническое и инфекционное сканирование загруженных файлов. Шаги выполняются строго друг за другом. Сбой на любом шаге прерывает пайплайн.

### 1.1. Шаг 1: Валидация формата (Apache Tika)
*   **Вход:** UUID версии файла (`fileVersionId`), S3-путь к временному объекту в `quarantine-bucket`.
*   **Что делает:** Считывает сигнатуру файла (magic bytes) непосредственно из S3-карантина, определяет реальный MIME-тип и сверяет его со списком разрешенных типов для данного `documentType` карточки.
*   **Выход:** Успешный вердикт соответствия MIME-типа либо генерация исключения валидации.

### 1.2. Шаг 2: Валидация размера (Max Size Boundary)
*   **Вход:** UUID карточки документа (`documentId`), размер файла из HEAD-запроса в S3.
*   **Что делает:** Сравнивает фактический размер загруженного в карантин объекта с жестко настроенной максимальной границей в байтах для конкретного типа документа.
*   **Выход:** Подтверждение корректности размера либо аварийное прерывание транзакции.

### 1.3. Шаг 3: Контроль целостности (SHA-256)
*   **Вход:** UUID версии файла, SHA-256 хэш-сумма, переданная клиентом при вызове `/complete`.
*   **Что делает:** Вычисляет SHA-256 хэш-сумму бинарного объекта в S3 и выполняет побайтовое сравнение с хэш-суммой, заявленной клиентом.
*   **Выход:** Подтверждение целостности файла либо отказ при несовпадении контрольных сумм.

### 1.4. Шаг 4: Антивирусное сканирование (ClamAV)
*   **Вход:** S3-путь к файлу в `quarantine-bucket`.
*   **Что делает:** Вызывает агент сканирования ClamAV для проверки бинарного тела файла на наличие вредоносного кода или сигнатур вирусов.
*   **Выход:** Статус `CLEAN` (файл чист). При обнаружении угрозы — статус `INFECTED` с немедленным удалением файла и изоляцией карточки.

## 2. Сервис работы с хранилищем и ссылками (`StorageIntegrationService`)

Сервис инкапсулирует прямое взаимодействие с объектным хранилищем S3 и отвечает за генерацию безопасных временных ссылок.

### 2.1. Метод: Перенос из карантина в основное хранилище
*   **Вход:** UUID версии файла (`fileVersionId`), S3-путь в `quarantine-bucket`.
*   **Что делает:** Копирует проверенный и чистый файл из `quarantine-bucket` в основное защищенное хранилище `main-bucket` с активацией защиты от перезаписи (Object Lock) и немедленно удаляет исходный объект из карантинной зоны.
*   **Выход:** S3-путь к постоянному объекту в `main-bucket`.

### 2.2. Метод: Генерация временной ссылки скачивания (Presigned Link)
*   **Вход:** Идентификатор пользователя (`userId`), роль субъекта, UUID карточки документа (`documentId`).
*   **Что делает:**
    1. Выполняет авторизационную проверку прав доступа (ACL/RBAC).
    2. Извлекает текущую чистую версию файла.
    3. Рассчитывает время жизни ссылки (TTL) исключительно на стороне сервера: **15 минут** для сотрудников ведомства (АРМ) и **30 минут** для заявителей (ЛК), в соответствии с BR-06 (MZD-18). Значения, переданные от клиента, полностью игнорируются.
    4. Генерирует подписанную ссылку через AWS Signature Version 4.
*   **Выход:** Строковый URL-адрес (`secureLinkRef`) с ограниченным временем жизни.

### 2.3. Метод: Генерация ссылки предпросмотра (Preview Link)
*   **Вход:** UUID карточки документа (`documentId`).
*   **Что делает:**
    1. Сверяет MIME-тип документа со списком разрешенных форматов для отображения в браузере (PDF, JPEG, PNG, TXT).
    2. Рассчитывает серверный TTL ссылки в **15 минут**.
    3. Переопределяет заголовки ответа S3 (`response-content-disposition = inline` и оригинальный MIME-тип) для предотвращения автоматического скачивания файла браузером.
*   **Выход:** Временная ссылка для встроенного рендеринга или признак невозможности предпросмотра (`previewAvailable: false`).

## 3. Сервис публикации документов заявителю (`DocumentPublishingService`)

Сервис обеспечивает передачу проверенных документов во внешний контур Личного Кабинета заявителя.

### 3.1. Метод: Публикация в ЛК
*   **Вход:** UUID карточки документа (`documentId`).
*   **Что делает:**
    1. Проверяет, находится ли карточка в статусе `READY` (все автопроверки успешно завершены).
    2. Проверяет тип документа: публикация запрещена для типа `SERVICE_RESOLUTION`.
    3. Сервер принудительно вычисляет профиль видимости (`visibilityProfile`), игнорируя любые входящие значения от внешних клиентов.
    4. Меняет статус `DocumentCard` на `PUBLISHED`.
    5. Регистрирует исходящее интеграционное событие `DocumentPublished` в Outbox-таблице для последующей доставки в `cabinet-service`.
*   **Выход:** Измененный объект `DocumentCard` в статусе `PUBLISHED` и сгенерированная запись события в Outbox.

## 4. Сервис квитанций доставки (`DeliveryReceiptService`)

Сервис предназначен для фиксации юридически значимых фактов взаимодействия заявителя с отправленными документами.

### 4.1. Метод: Регистрация факта (Доставка / Просмотр / Скачивание)
*   **Вход:** UUID документа (`documentId`), UUID получателя (`recipientId`), тип регистрируемого действия (`DELIVERED`, `OPENED`, `DOWNLOADED`).
*   **Что делает:**
    1. Формирует композитный ключ защиты от дублей: `[documentId, recipientId, receiptType]`.
    2. Выполняет проверку линейности воронки: событие `DOWNLOADED` не может быть создано раньше `DELIVERED`.
    3. Выполняет попытку вставки новой записи в таблицу `DeliveryReceipt`. При коллизии ключа уникальности срабатывает инструкция `ON CONFLICT DO NOTHING`, предотвращая дублирование данных.
    4. Сервис работает по WORM-модели: операции изменения (`UPDATE`) и удаления (`DELETE`) физически заблокированы на уровне СУБД. Квитанции не удаляются даже при отзыве публикации документа.
*   **Выход:** Запись о квитанции в БД либо игнорирование дублирующего запроса без изменения данных.
