# MZD-36: Сценарии публикации и предпросмотра документов

Настоящая спецификация определяет логику публикации документов в Личный Кабинет (ЛК) заявителя, процесс генерации защищённых временных ссылок для скачивания и сценарий интерактивного предпросмотра файлов в браузере. Действия ссылок соответствуют enum `SecureLinkAction` (SPEC-13 §3.4): `VIEW` и `DOWNLOAD`.

## 1. Сценарий публикации документа заявителю

Процесс публикации переводит документ из внутреннего контура ведомства в публичный контур ЛК (переход `APPROVED → PUBLISHED`, REQ-08-010).

### Системный поток публикации:
1. **Инициация:** смежный сервис (например, report-service) отправляет POST-запрос на эндпоинт `/internal/v1/documents/{documentId}/publish`.
2. **Валидация бизнес-правил:**
   * Статус карточки должен быть строго `APPROVED` — иначе `409 Conflict` (MZD-25).
   * Тип документа: для `SERVICE_RESOLUTION` запрос мгновенно отклоняется жёстким guard'ом — `403 Forbidden` (`SecurityAccessDeniedException`, MZD-22 §2.5). Для прочих типов, у которых `PUBLISHED` исключён из графа (`REQUEST_MATERIAL`, `EXPERT_CARD`, `COMPARATIVE_PROTOCOL`), — `422 Unprocessable Entity`.
   * Актуальная версия файла — в статусе `CLEAN` (иначе `409`, ФТ-07).
3. **Применение профиля видимости:**
   * Сервер извлекает из БД `visibilityProfile`, сохранённый при создании карточки; публикация допустима только для `APPLICANT_RESULT` / `APPLICANT_REQUEST_MATERIAL` (`INTERNAL` → `422`).
   * **Строгое правило:** значение профиля видимости рассчитывается и контролируется исключительно сервером. Любые попытки клиента передать или переопределить профиль в теле запроса `/publish` игнорируются или приводят к `400 Bad Request`.
4. **Фиксация статуса:**
   * Статус `DocumentCard` переводится в `PUBLISHED`, генерируется новый `publication_id` (акт публикации, MZD-34 §2).
   * Сервис публикует событие `DocumentPublished` (payload по MZD-40: `documentId`, `regNumber`, `applicantId`, `visibilityProfile`, `secureLinkRef`-идентификатор, `publishedAt`) для нотификации `cabinet-service`.

## 2. Сценарий выдачи временной ссылки (Secure Link, действие `DOWNLOAD`)

Для скачивания файла клиенту (АРМ сотрудника или ЛК заявителя) предоставляется временная подписанная ссылка (presigned URL), ведущая напрямую в S3-хранилище `main-bucket`. Ссылки выдаются только на версии в статусе `CLEAN` (MZD-33 §4).

### Системный поток генерации ссылки:
1. **Запрос ссылки:** пользователь нажимает «Скачать»; клиент отправляет GET-запрос на `/api/v1/documents/{documentId}/download-link`.
2. **Валидация прав доступа (ACL):**
   * Сервер авторизует пользователя и проверяет его права на чтение документа (RBAC/ABAC — `403` при отказе).
   * Проверяется профиль видимости: заявитель не может получить ссылку на документ с `visibilityProfile = INTERNAL`; для `SERVICE_RESOLUTION` создание ссылки заблокировано полностью (MZD-46, F1.4).
3. **Расчёт срока жизни ссылки (TTL):**
   * **Строгое правило:** TTL рассчитывается исключительно сервером на основе роли пользователя и настроек безопасности. Значения TTL, переданные клиентом, полностью игнорируются.
   * Для внутренних сотрудников ведомства (АРМ): TTL = **15 минут**.
   * Для внешних пользователей (ЛК заявителя): TTL = **30 минут**.
4. **Генерация Presigned URL:**
   * Сервис обращается к S3 по протоколу AWS Signature Version 4, передавая путь к объекту, HTTP-метод (`GET`) и рассчитанный сервером `X-Amz-Expires` (равный TTL).
   * Клиенту возвращается JSON с защищённым URL. По истечении TTL ссылка автоматически аннулируется на уровне S3.
   * Сырой presigned URL живёт только в синхронном REST-ответе: в события и брокер он не попадает — там только `secureLinkRef`-идентификатор (MZD-40 §2, MZD-46 F2).
5. **Квитанция:** успешная выдача ссылки заявителю фиксируется квитанцией `DOWNLOADED` (MZD-34).

## 3. Сценарий предпросмотра в браузере (Preview Engine, действие `VIEW`)

Интерактивный предпросмотр позволяет ознакомиться с содержимым документа непосредственно в браузере без принудительного скачивания файла (REQ-08-008).

### Системный поток предпросмотра:
1. **Запрос на просмотр:** пользователь нажимает иконку просмотра («глаз»); клиент отправляет GET-запрос `/api/v1/documents/{documentId}/preview-link`.
2. **Валидация MIME-типа (Preview Whitelist):**
   * Сервер проверяет реальный MIME-тип файла, определённый Apache Tika на этапе загрузки.
   * Если формат поддерживает отображение в браузере (PDF, JPEG, PNG, TXT) — генерация ссылки разрешается.
   * Для неподдерживаемых форматов (архивы ZIP/RAR, чертежи CAD, исполняемые файлы) возвращается флаг `previewAvailable: false`, кнопка просмотра в UI блокируется.
3. **Генерация ссылки предпросмотра:**
   * Сервер формирует presigned-ссылку в S3 с параметром `ResponseHeaderOverrides`:
     * `response-content-disposition = inline` — открыть во встроенном просмотрщике, а не скачивать;
     * `response-content-type` = оригинальный MIME-тип файла (например, `application/pdf` или `image/png`).
4. **Рендеринг:** фронтенд принимает временную ссылку с флагом `inline` и рендерит документ во внутреннем iframe или специализированном PDF-вьювере. TTL ссылки предпросмотра также рассчитывается сервером и равен **15 минутам**.
5. **Квитанция:** запуск предпросмотра заявителем фиксируется квитанцией `OPENED` (MZD-34).
