# MZD-23: Архитектурные бизнес-правила интеграционного домена

## 1. Решение по стратегии Нумерации: Gapless vs Gaps (Перенос в SPEC-16)

Окончательный выбор между безразрывной (`Gapless`) и производительной (`Gaps allowed`) нумерацией является бизнес-требованием зоны делопроизводства и возвращен в **SPEC-16** для согласования с заказчиком. Ниже представлены два архитектурных варианта с оценкой последствий:

### Вариант А: Безразрывная нумерация (Gapless)
* **Механизм:** Требует строгого пессимистического захвата блокировки на строку текущего счетчика (`SELECT FOR UPDATE по строке журнала` в PostgreSQL) в рамках транзакции создания документа. В случае отката транзакции номер не теряется, разрывов нет.
* **Последствия:** Создает конкурентное узкое горлышко (bottleneck) при одновременной массовой регистрации документов в один и тот же журнал. Потоки будут ожидать освобождения блокировки строки счетчика, что существенно снижает пропускную способность (TPS) системы.

### Вариант Б: Нумерация с допущением разрывов (Gaps allowed)
* **Механизм:** Использование высокопроизводительных последовательностей (Sequences) базы данных или распределенных генераторов ID. Номер резервируется до коммита транзакции. При отказе транзакции номер сгорает (образуется разрыв).
* **Последствия:** Максимальное быстродействие, отсутствие взаимных блокировок потоков, соответствие высоким целевым показателям отзывчивости системы.

## 2. Идемпотентность и обработка сбоев по НФТ
* **Дедупликация:** Входящие сообщения и запросы валидируются через уникальный `idempotencyKey` на прикладном уровне.
* **Параметры повторных попыток (Retry Policy) согласно НФТ:**
  * **Брокер сообщений (Kafka Consumer):** Настроено **5 попыток** повторной обработки с экспоненциальной задержкой (Backoff), после чего сообщение отправляется в карантинный топик.
  * **Синхронные вызовы (REST-клиент / WebClient):** Настроено **3 попытки** выполнения запроса к смежным сервисам при сетевых сбоях (ошибки 5xx, ConnectTimeout).
* **Хранение Dead Letter Queue (DLQ):** Все аварийные сообщения, попавшие в `document-service-dlq`, гарантированно сохраняются в СУБД/хранилище на срок **не менее 14 дней** для обеспечения возможности ручного разбора, исправления и повторного запуска (Replay) администраторами системы.

## 3. Правила кросс-документального связывания
* Валидация связей выполняется через поле `linkType` (SPEC-13). 
* Проверка бизнес-логики: При сохранении карточки документа типа `Сравнительный протокол` сервис проверяет обязательное наличие в таблице связей записи с `linkType = PRIMARY_EXPERTISE`. При отсутствии связи транзакция прерывается с ошибкой `BusinessValidationException`, что устраняет любые внутренние противоречия в документации.
