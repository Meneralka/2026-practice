# Процедура разбора недоставленных сообщений (DLQ) · document-service

**Тип документа:** эксплуатационный регламент (runbook) для администратора
**Область:** обнаружение, разбор, исправление и перезапуск обработки сообщений, попавших в DLQ
**Основания:** SPEC-09 (@ArtemKvvs) — outbox, DLQ, ручной replay; стейкхолдер-документ р.10 (5 попыток, состав DLQ-сообщения, retention); SPEC-14 (envelope); MZD-46 (безопасность)

---

## 1. Как сообщение попадает в DLQ

```
Событие в топике (*.events.v1)
        │
        ▼
  Consumer обрабатывает
        │
        ├── успех ──────────────────────► обработано, DLQ не задействован
        │
        └── ошибка
              │
              ▼
        Retry: до 5 попыток, экспоненциальная задержка
              │
              ├── успех на попытке 2..5 ─► обработано, DLQ не задействован
              │
              └── 5 попыток исчерпаны
                        │
                        ▼
                  DLQ-топик ◄── сообщение ждёт ручного разбора
```

**Важно:** попадание в DLQ — это **не** потеря сообщения. Событие сохранено целиком и ждёт администратора. Потеря возможна только если DLQ не разобран до истечения retention (**≥ 14 календарных дней**).

## 2. Состав DLQ-сообщения

Каждое сообщение в DLQ содержит (стейкхолдер-документ р.10):

| Поле | Назначение при разборе |
|---|---|
| **Исходное сообщение** | Полный envelope (9 полей): `eventId`, `eventType`, `eventVersion`, `occurredAt`, `producer`, `correlationId`, `causationId`, `businessKey` (ID заявки, не документа), `payload` |
| `errorClass` | Класс исключения — основной признак для классификации причины |
| `errorMessage` | Текст ошибки |
| `stackTraceHash` | Хэш стектрейса — группировка однотипных сбоев |
| `consumer` | Какой consumer не смог обработать |
| `failedAt` | Время последней неудачной попытки |
| `retryCount` | Сколько попыток исчерпано (должно быть 5) |
| `correlationId` | **Ключ для сквозного расследования** — по нему поднимаются логи и трейсы всей цепочки |

## 3. Обнаружение

### 3.1. Автоматическое (основной путь)

| Источник | Что смотреть |
|---|---|
| **Prometheus / Grafana** | Счётчик сообщений в DLQ. Ненулевой рост = требуется разбор |
| **Alert** | Срабатывание при появлении сообщений в DLQ |
| **Grafana (Loki/ELK)** | Лог-записи об исчерпании retry |
| **Consumer lag** | Косвенный признак: растущий лаг + DLQ = систематический сбой, а не единичный |

> ⚠️ **Пробел:** конкретные алерты и пороги (при скольких сообщениях в DLQ будить администратора) не определены. См. раздел 9.

### 3.2. Ручное

Периодический просмотр DLQ-топиков. Обязателен, если алертинг ещё не настроен.

## 4. Триаж: определить масштаб

Прежде чем чинить — понять, что это.

| Признак | Диагноз | Действие |
|---|---|---|
| 1–2 сообщения, разные `stackTraceHash` | Единичные сбои | Разбор по одному (раздел 5) |
| Много сообщений, **один** `stackTraceHash` | Систематический дефект | Сначала устранить причину, потом replay пачкой |
| Много сообщений, один `consumer`, разные хэши | Проблема самого consumer'а / его зависимости | Проверить здоровье consumer'а и смежного сервиса |
| Рост в реальном времени | **Активная авария** | Эскалация, не начинать replay до устранения |

**Правило:** не запускать replay, пока причина не устранена. Сообщение снова пройдёт 5 попыток и вернётся в DLQ, только с новым `failedAt`.

## 5. Разбор одного сообщения

### Шаг 1. Извлечь сообщение из DLQ

Прочитать сообщение, зафиксировать: `eventId`, `correlationId`, `errorClass`, `consumer`, `retryCount`.

### Шаг 2. Восстановить контекст по `correlationId`

Это ключевой шаг. `correlationId` сквозной — по нему собирается вся цепочка:

| Инструмент | Что даёт |
|---|---|
| **Jaeger / OpenTelemetry** | Полный трейс: исходный REST-запрос → publish в Kafka → попытки обработки consumer'ом. Видно, где именно порвалось |
| **Grafana (Loki/ELK)** | Логи всех участников цепочки с тем же `correlationId` |
| `causationId` | Событие-предшественник — если сбой каскадный, ведёт к первопричине |
| `businessKey` | Бизнес-контекст (какая заявка / документ) |

### Шаг 3. Классифицировать причину

| Класс | Типовые признаки | Устранение |
|---|---|---|
| **A. Временная недоступность** | `errorClass` — таймаут/connection refused; смежный сервис лежал | Убедиться, что сервис поднят → replay без правок |
| **B. Несовместимость схемы** | Ошибка десериализации; `eventVersion` не поддерживается consumer'ом | Обновить consumer или Schema Registry → replay |
| **C. Дефект в данных события** | Валидация payload; отсутствует обязательное поле | ⚠️ См. раздел 6 — **сообщение править нельзя** |
| **D. Дефект кода consumer'а** | NPE, логическая ошибка | Исправить код → деплой → replay |
| **E. Нарушение бизнес-правила** | Документ в недопустимом статусе, нет обязательной связи | Привести состояние в порядок → replay; либо признать событие невалидным |
| **F. Отравленное сообщение** | Сообщение ломает consumer при любых условиях | Не ретраить. Разбор с бэкендером, при необходимости — списание (раздел 7) |

### Шаг 4. Устранить причину

По классификации из шага 3. **Не переходить к replay, пока причина не устранена.**

### Шаг 5. Replay

Отправить сообщение обратно в исходный топик на повторную обработку.

**Что защищает от дублей:** consumer идемпотентен и хранит обработанные `eventId` (стейкхолдер-документ р.6.1). Даже если сообщение частично отработало до сбоя, повторное действие не выполнится.

**Порядок:** при replay нескольких сообщений одного ключа (`businessKey` / `documentId`) отправлять **в исходном порядке** `occurredAt`. Порядок гарантируется только внутри ключа; между разными ключами порядок не важен.

### Шаг 6. Верификация

| Проверка | Ожидание |
|---|---|
| Сообщение обработано | В логах consumer'а успех по тому же `correlationId` |
| Не вернулось в DLQ | Счётчик DLQ уменьшился на 1 |
| Побочный эффект однократен | Одна запись, не две (идемпотентность отработала) |
| Бизнес-состояние корректно | Документ/связь/статус в ожидаемом виде по `businessKey` |
| Трейс полный | В Jaeger цепочка по `correlationId` завершена без разрывов |

### Шаг 7. Зафиксировать

| Что записать | Зачем |
|---|---|
| `eventId`, `correlationId`, класс причины, что сделано, когда | Аудит + основание для устранения системной причины |
| `stackTraceHash` | Если повторится — сразу видно, что это уже известный дефект |

## 6. Ограничение: сообщение править нельзя

**Формулировка задачи предполагает, что администратор «исправляет» сообщение. Это требует уточнения.**

Событие в Kafka — **неизменяемый факт**: «это произошло». Отредактировать его и переотправить означает подделать историю: изменённое событие будет отличаться от того, что реально опубликовал producer, при том же `eventId`. Это ломает:

- **аудит** — журнал перестаёт отражать реальность;
- **идемпотентность** — тот же `eventId` с другим содержимым (SPEC-09 прямо запрещает такое для REST: тот же `idempotencyKey` + другое тело → 409);
- **воспроизводимость** — расследование инцидента по трейсу становится невозможным.

**Поэтому «исправление» — это исправление причины, а не сообщения:**

| Что администратор реально делает | Что он НЕ делает |
|---|---|
| Поднимает упавший сервис | Не редактирует payload |
| Обновляет consumer / схему | Не подменяет поля |
| Правит состояние бизнес-объекта | Не пересобирает envelope |
| Заказывает фикс кода бэкендеру | Не меняет `eventId` |
| Инициирует **новое** корректирующее событие от producer'а | Не «чинит» старое |

Если данные в событии действительно битые (класс C) — правильный путь: **producer публикует новое корректное событие**, а битое списывается (раздел 7). Producer у нас — владелец домена (для `document.events.v1` — команда document-service).

> ⚠️ **Требует подтверждения:** если под «исправляет» в задаче имелась в виду именно правка payload — это архитектурное решение, которое надо согласовать с @ArtemKvvs, а не регламент администратора. См. раздел 9.

## 7. Списание сообщения (когда replay невозможен)

Применяется для классов **C** и **F**, когда событие невозможно обработать в принципе.

| Шаг | Действие |
|---|---|
| 1 | Согласовать с владельцем домена (producer) — списание, а не replay |
| 2 | Зафиксировать: `eventId`, `correlationId`, `businessKey`, причина списания, кто согласовал |
| 3 | Определить, требуется ли компенсация: нужно ли producer'у опубликовать корректирующее событие |
| 4 | Убедиться, что бизнес-состояние консистентно **без** этого события |
| 5 | Удалить/пометить сообщение как списанное |

**Списание без фиксации причины запрещено.** Иначе теряется факт, что событие было и не отработало.

## 8. Безопасность при разборе (MZD-46)

| Правило | Обоснование |
|---|---|
| DLQ содержит **полные payload'ы** событий, включая `businessKey`, `applicantId`, ссылки на документы | Доступ к DLQ = доступ к бизнес-данным. Ограничить кругом администраторов |
| Не выгружать содержимое DLQ в тикеты/чаты в сыром виде | Утечка метаданных. Ссылаться на `eventId` / `correlationId`, а не копировать payload |
| Не логировать payload при разборе | Дублирование данных за пределы контура |
| Пользователю ошибка показывается **без** технических деталей | Стейкхолдер-документ р.10; детали — только в логах по `correlationId` |

> ⚠️ **Зависимый риск:** если `secureLinkRef` в payload `DocumentPublished` окажется сырым presigned URL (открытый вопрос к @ArtemKvvs), то **DLQ хранит рабочие ссылки на файлы 14 дней**. Тогда доступ к DLQ = прямой доступ к документам в обход прав. До закрытия этого вопроса DLQ следует считать хранилищем чувствительных данных.

## 9. Незакрытые пункты

| # | Пробел | Почему важно | Кому |
|---|---|---|---|
| 1 | **Инструмент доступа администратора к DLQ не определён** (CLI `kafka-console-consumer` / UI типа AKHQ / Kafdrop / собственный endpoint) | Без него процедура не операционализируется: неясно, чем читать и чем переотправлять | DevOps / @ArtemKvvs |
| 2 | **Механизм replay не определён** — ручной re-publish, admin-endpoint в сервисе, или отдельная утилита | SPEC-09 фиксирует «ручной replay», но не то, чем он выполняется | @ArtemKvvs |
| 3 | **Именование DLQ-топиков не зафиксировано** (`<topic>.DLT`? отдельный топик на consumer group?) | Влияет на навигацию и на права доступа | @ArtemKvvs / DevOps |
| 4 | **Алерты и пороги** — при каком количестве сообщений в DLQ поднимать тревогу, кому и по какому каналу | Без этого DLQ обнаруживается случайно; при retention 14 дней — риск реальной потери | DevOps |
| 5 | **«Исправление сообщения»** — подтвердить, что имеется в виду устранение причины, а не правка payload (раздел 6) | Правка payload ломает аудит и идемпотентность; если это всё же требуется — нужно архитектурное решение | @ArtemKvvs |
| 6 | **Роль/права администратора DLQ** — кто именно допущен, как авторизуется | DLQ = чувствительные данные (раздел 8) | DevOps / безопасность |
| 7 | **`secureLinkRef` в payload** — идентификатор или сырой presigned URL | Определяет, является ли DLQ хранилищем рабочих ссылок на файлы | @ArtemKvvs |
| 8 | **Delayed retry topic** — стейкхолдер-документ допускает его для временных ошибок, но условия перехода в него vs сразу в DLQ не описаны | Влияет на классификацию класса A | @ArtemKvvs |

## 10. Краткая карточка (для дежурного)

```
1. АЛЕРТ: сообщения в DLQ
        ↓
2. ТРИАЖ: сколько? один stackTraceHash или разные?
   └─ растёт в реальном времени → ЭСКАЛАЦИЯ, replay не начинать
        ↓
3. КОНТЕКСТ: взять correlationId → Jaeger + Grafana
        ↓
4. КЛАССИФИКАЦИЯ: A временная / B схема / C данные /
                  D код / E бизнес-правило / F отравленное
        ↓
5. УСТРАНИТЬ ПРИЧИНУ  ← НЕ править сообщение
        ↓
6. REPLAY (в порядке occurredAt внутри одного ключа)
        ↓
7. ВЕРИФИКАЦИЯ: обработано? не вернулось? эффект однократен?
        ↓
8. ЗАФИКСИРОВАТЬ: eventId, correlationId, причина, действие
```

**Помнить:**
- Retention DLQ **≥ 14 дней** — после этого сообщение теряется безвозвратно.
- Consumer **идемпотентен** — replay не создаёт дублей.
- Сообщение **не редактируется** — исправляется причина.
- DLQ содержит **бизнес-данные** — обращаться как с чувствительными.
