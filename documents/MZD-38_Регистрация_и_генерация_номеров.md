# MZD-38: Регистрация документов и генерация номеров

Регистрация ведётся в журналах по типам документов (`RegistryCode`, SPEC-13 §3.8, REQ-08-006): один журнал на тип, номер уникален в пределах пары (журнал, год), формат вида `RM-2026-000123`.

**Момент присвоения:** регистрационный номер присваивается при переходе карточки в `ACCEPTED` (документ включён в материалы дела) — поэтому событие `DocumentAccepted` уже несёт `regNumber` (MZD-40). Для `SERVICE_RESOLUTION`, в графе которой статус `ACCEPTED` отсутствует (MZD-22 §2.5), номер присваивается при переходе в `APPROVED`.

## 1. Сценарий генерации с защитой от дублей
* Процедура резервирования и присвоения номера реализуется через БД-последовательность PostgreSQL (Sequence) — по одной на пару (журнал, год).
* Точка сериализации при массовой загрузке — пессимистическая блокировка строки журнала (`SELECT FOR UPDATE`), зафиксировано решением Q-2 (close-questions).
* **Разрывы в нумерации допустимы (gaps allowed, Q-2):** юридического требования безразрывности в ТЗ нет; откат транзакции может «сжечь» номер, и это штатно. Если заказчик впоследствии потребует gapless-нумерацию, переход локализован в пакете `registry/`.

## 2. Поведение при повторном запросе (Идемпотентность)
* Если у карточки `DocumentCard` поле `reg_number` уже заполнено, повторный вызов метода регистрации возвращает **тот же самый номер** и статус `200 OK` без обращения к генератору последовательностей.
* Идемпотентность гарантируется сервером (MZD-21 §2): вход `GenerateNumberRequest` содержит `documentType`, `journalId`, `idempotencyKey`; повторный запрос с идентичным `idempotencyKey` в рамках одной пары (`documentType`, `journalId`) возвращает сохранённый номер из БД без повторного инкремента.
* Конфликт `idempotencyKey` (тот же ключ, другое тело запроса) → `409 Conflict` (единый код, MZD-25).
