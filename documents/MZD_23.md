# MZD-23: Архитектурные бизнес-правила интеграционного домена

## 1. Решение по стратегии Нумерации: Gaps allowed (закрыто)

Зафиксирован выбор в пользу производительной нумерации: **Gaps allowed**. Разрывы номеров допустимы и не считаются дефектом; уникальность и порядок обеспечиваются `Sequence` PostgreSQL, а на прикладном уровне повтор возвращает тот же номер по `idempotencyKey`.

* **Механизм:** Высокопроизводительная последовательность (`Sequence`) БД. Номер резервируется до коммита транзакции. При откате транзакции номер сгорает (образуется разрыв) — это не является ошибкой.
* **Идемпотентность:** Повторный запрос с идентичным `idempotencyKey` возвращает ранее выданный номер без повторного инкремента.
* **Почему не Gapless:** Строгий пессимистический захват блокировки строки счетчика (`SELECT FOR UPDATE`) создаёт конкурентное узкое горлышко при массовой регистрации и снижает TPS. Юридического требования безразрывности в ТЗ нет.
* **Путь отступления:** Если заказчик впоследствии потребует безразрывность, переход на `SELECT FOR UPDATE` локализован в пакете `registry/` и не затрагивает остальной домен.

## 2. Идемпотентность и обработка сбоев по НФТ
* **Дедупликация:** Входящие сообщения и запросы валидируются через уникальный `idempotencyKey` на прикладном уровне.
* **Параметры повторных попыток (Retry Policy) согласно НФТ:**
  * **Брокер сообщений (Kafka Consumer):** Настроено **5 попыток** повторной обработки с экспоненциальной задержкой (Backoff), после чего сообщение отправляется в карантинный топик.
  * **Синхронные вызовы (REST-клиент / WebClient):** Настроено **3 попытки** выполнения запроса к смежным сервисам при сетевых сбоях (ошибки 5xx, ConnectTimeout).
* **Хранение Dead Letter Queue (DLQ):** Все аварийные сообщения, попавшие в `document-service-dlq`, гарантированно сохраняются в СУБД/хранилище на срок **не менее 14 дней** для обеспечения возможности ручного разбора, исправления и повторного запуска (Replay) администраторами системы.

## 3. Правила кросс-документального связывания
* Валидация связей выполняется через поле `linkType` (SPEC-13). 
* Проверка бизнес-логики: При сохранении карточки документа типа `Сравнительный протокол` сервис проверяет обязательное наличие в таблице связей записи с `linkType = PRIMARY_EXPERTISE`. При отсутствии связи транзакция прерывается с ошибкой `BusinessValidationException`, что устраняет любые внутренние противоречия в документации.
