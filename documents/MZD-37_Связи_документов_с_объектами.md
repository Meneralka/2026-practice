# MZD-37: Спецификация связей между документами и объектами

Спецификация описывает механизм связывания документов с бизнес-объектами системы в соответствии с MZD-21 и SPEC-13.

## 1. Модель данных (Many-to-Many)
Согласно SPEC-13 и MZD-21, связи организуются по схеме: один документ может быть привязан к нескольким бизнес-объектам, и один объект содержит множество документов.
Связи хранятся в таблице `document_link`:
* `id` (UUID, Primary Key)
* `document_id` (UUID, Foreign Key, `document_card.id`) — ID документа.
* `target_id` (UUID) — ID целевого бизнес-объекта (заявка, дело, отчет и т.д.).
* `link_type` (VARCHAR / Enum) — тип связи (строго соответствует `LinkType` из `common/enums/`).
* `created_at` (TIMESTAMP)

Отдельного поля `relatedEntityType` на `DocumentCard` нет: тип целевого объекта однозначно выводится из `linkType` (решение по конфликту MZD-5/MZD-6, зафиксировано в SPEC-13 §2.2).

## 2. Типы связей (`linkType`)
Допускается строго 7 системных типов связей (SPEC-13 §2.2, REQ-08-005 + PRIMARY_EXPERTISE):
1. `REQUEST` — Заявка (владелец: request-service)
2. `EXPERTISE_CASE` — Дело экспертизы (expertise-service)
3. `EXPERT_CARD` — Экспертная карта (expertise-service)
4. `REVIEW` — Проверка (review-service)
5. `PRIMARY_EXPERTISE` — Первичная экспертиза, дело первичной экспертизы (необходима для кросс-документальной валидации)
6. `REPORT` — Отчет (report-service)
7. `ROUTE` — Маршрут ЭДО (workflow-service)

## 3. Защита от дублирования
* **На уровне БД:** Навешивается уникальный составной индекс:
  `UNIQUE(document_id, link_type, target_id)`
* **На уровне бизнес-логики:** Перед записью выполняется проверка на существование связи. При обнаружении дубля вызов завершается идемпотентно с возвратом существующей записи (`200 OK`) во избежание генерации ошибок при повторных сетевых вызовах.
* **Событие:** Успешное создание новой связи публикует `DocumentLinkCreated` (payload: `documentId`, `targetId`, `linkType`); идемпотентный повтор событие не порождает.
