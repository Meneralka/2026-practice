# MZD-31: Обработка исключительных ситуаций и ошибок при загрузке документов

Настоящая спецификация определяет поведение системы, статусные последствия и текстовые нотификации для пользователя при возникновении ошибок на этапах инициации, загрузки и технической валидации файлов. Статусы карточки — по MZD-32 (SPEC-13), статусы версии файла — по MZD-16 / MZD-33, коды причин — `ReasonCode` (SPEC-13 §3.3).

## 1. Общая матрица обработки исключений

| Код ситуации | Описание ошибки | Точка фиксации | Статус `DocumentCard` | Статус `FileVersion` | Реакция системы / Бизнес-логика | Код ответа API |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **ERR-01** | Обнаружен вирус (ClamAV) | Асинхронный конвейер | `DRAFT` (возврат из `UNDER_REVIEW`) | `INFECTED` | Удаление бинарного файла из S3-карантина, флаг `quarantined: true`, событие `DocumentTechnicalCheckFailed` (`reasonCode = ANTIVIRUS_INFECTED`). Бизнес-операции с карточкой блокируются. | `409 Conflict` (на попытку получить ссылку / привязать / двинуть статус) |
| **ERR-02** | Неверный тип файла (Apache Tika) | Асинхронный конвейер | `DRAFT` (возврат из `UNDER_REVIEW`) | `INVALID` | Сравнение реального MIME-типа по magic bytes с вайтлистом. `reasonCode = MIME_TYPE_NOT_ALLOWED`, дальнейший маппинг блокируется. | `409 Conflict` |
| **ERR-03-A** | Превышение размера при `/init` | Синхронно на фазе `/init` | Не создается | Не создается | Валидация `fileSizeBytes` по конфигурационной матрице. Отказ до выделения места. | `413 Payload Too Large` |
| **ERR-03-B** | Превышение размера при заливке в S3 | Хранилище RustFS | `DRAFT` | `NEW` | Отклонение S3-запроса на уровне бакета благодаря `content-length-range` в выданном POST Policy. | `413 Payload Too Large` (от S3) |
| **ERR-03-C** | Превышение размера при `/complete` | Синхронно на фазе `/complete` | `DRAFT` | `INVALID` | Финальный HEAD-запрос к S3 выявляет расхождение с заявленным размером. Сессия аннулируется, `reasonCode = FILE_SIZE_EXCEEDED`. | `413 Payload Too Large` |
| **ERR-04** | Недоступность антивируса (ClamAV) | Асинхронный конвейер | `UNDER_REVIEW` | `CHECKING` (зависает) | Ретраи по политике экспоненциального бэкоффа (5 попыток / 10 минут). По исчерпании — алерт в Sentry/Opsgenie и ручной разбор ИБ; при подтверждённой невозможности проверки версия переводится в `INVALID` (`reasonCode = ANTIVIRUS_SCAN_FAILED`). | `409 Conflict` (на попытку публикации / скачивания / привязки) |
| **ERR-05-A** | Недоступность RustFS при `/init` | Синхронно на фазе `/init` | Не создается | Не создается | Отказ при резервировании объекта в S3-карантине. Транзакция БД откатывается. | `503 Service Unavailable` |
| **ERR-05-B** | Недоступность RustFS при `/complete` | Синхронно на фазе `/complete` | `DRAFT` | `NEW` | Невозможно выполнить HEAD-запрос или транзитные операции. Ретраи на уровне API-шлюза. | `503 Service Unavailable` |
| **ERR-06** | Клиентская отмена загрузки | Асинхронно / Таймаут | `DRAFT` (или удаление пустой карточки) | `NEW` (сессия аннулирована) | Пользователь прерывает процесс в UI или истекает TTL сессии загрузки (2 часа). Асинхронный воркер очищает незавершённые S3-multipart-загрузки (MZD-33 §4). | `404 Not Found` (при попытке завершить `/complete`) |

Конвенция кодов: `413` — нарушение лимита размера; `409` — конфликт состояния (файл не в `CLEAN`, ФТ-07); `503` — недоступность инфраструктуры; `404` — аннулированная/несуществующая сессия. См. также MZD-32 §5.

## 2. Детальные сценарии, реакция системы и UI-сообщения

### ERR-01: Обнаружен вирус (ClamAV)
* **Системный поток:**
  1. Конвейер проверок сканирует файл в `quarantine-bucket`.
  2. Обнаружена сигнатура вируса → версия переводится в `INFECTED`, карточка откатывается в `DRAFT`, выставляется `quarantined: true`.
  3. Сервис публикует событие `DocumentTechnicalCheckFailed` (`reasonCode = ANTIVIRUS_INFECTED`).
  4. Физический файл в `quarantine-bucket` немедленно уничтожается.
* **Реакция интерфейса (UI):** карточка файла подсвечивается красным. Блокируются кнопки «Скачать» и «Использовать». Возле названия файла — иконка предупреждения ИБ.
* **Текст сообщения для пользователя:**
  > **Файл не прошел проверку безопасности**
  > В документе `{fileName}` обнаружены потенциально опасные объекты или вредоносный код. Файл был автоматически удален и изолирован. Пожалуйста, проверьте ваш персональный компьютер антивирусом и загрузите очищенную версию файла.

### ERR-02: Неверный тип файла (Apache Tika)
* **Системный поток:**
  1. После постановки файла на конвейер анализатор Apache Tika парсит заголовок файла (magic bytes).
  2. Выявлен реальный MIME-тип, не входящий в вайтлист для `documentType` (например, `application/x-dosexec` при заявленном расширении `.pdf`).
  3. Конвейер останавливает обработку, файл удаляется из карантина.
  4. Версия переводится в `INVALID` (`reasonCode = MIME_TYPE_NOT_ALLOWED`), карточка возвращается в `DRAFT`.
* **Реакция интерфейса (UI):** поле загрузки сбрасывается, отображается системное окно с ошибкой валидации формата.
* **Текст сообщения для пользователя:**
  > **Неподдерживаемый формат файла**
  > Файл `{fileName}` не может быть загружен. Реальное содержимое файла не соответствует расширению `{fileExtension}`. Пожалуйста, пересохраните документ в оригинальном формате (разрешены: PDF, JPEG, PNG) и повторите попытку.

### ERR-03: Превышение размера файла (Max Size Boundary)
* **Системный поток (превентивный и реактивный):**
  * *Сценарий А:* пользователь отправляет форму с размером больше лимита — `/init` сразу возвращает `413 Payload Too Large`.
  * *Сценарий Б:* обходной путь через подмену API-запроса напрямую в S3 блокируется политикой POST Policy (`content-length-range`) — хранилище возвращает ошибку загрузки.
  * *Сценарий В:* если HEAD-запрос на `/complete` выявляет расхождение с лимитом, транзакция прерывается, версия помечается `INVALID` (`reasonCode = FILE_SIZE_EXCEEDED`).
* **Реакция интерфейса (UI):** полоса загрузки окрашивается в красный с прерыванием процесса.
* **Текст сообщения для пользователя:**
  > **Превышен максимальный размер файла**
  > Размер файла `{fileName}` составляет `{actualSize} МБ`, что превышает установленный лимит в `{maxSize} МБ` для данного типа документа. Пожалуйста, сожмите файл или оптимизируйте его разрешение перед повторной загрузкой.

### ERR-04: Недоступность антивирусного сканера (ClamAV)
* **Системный поток:**
  1. На шаге `/complete` файл успешно загружен, версия — `UPLOADED` → `CHECKING`, карточка — `UNDER_REVIEW`.
  2. Конвейер не может связаться с ClamAV (таймаут сети или падение инстанса).
  3. Запускается механизм повторных попыток (Exponential Backoff, 5 ретраев в течение 10 минут).
  4. По исчерпании лимита — критический алерт в мониторинг (Sentry/Opsgenie), кейс уходит на ручной разбор ИБ; карточке запрещено движение по бизнес-цепочке (`409`, Fail-Secure — MZD-35 §3).
* **Реакция интерфейса (UI):** напротив файла — лоадер со статус-текстом «Проверка безопасности задерживается».
* **Текст сообщения для пользователя:**
  > **Проверка безопасности занимает больше времени, чем обычно**
  > Ваш документ успешно загружен и находится в очереди на проверку. Мы завершим её в фоновом режиме. Вам не нужно загружать файл повторно — статус обновится автоматически в течение нескольких минут.

### ERR-05: Недоступность объектного хранилища (RustFS)
* **Системный поток:**
  * *Сценарий А (при инициации):* при `/init` сервис не может связаться с RustFS для резервирования пути — запрос завершается `503 Service Unavailable`, запись в БД не фиксируется.
  * *Сценарий Б (при завершении):* при `/complete` S3 недоступен для HEAD-запроса — верификация приостанавливается, клиенту возвращается `503`.
* **Реакция интерфейса (UI):** модальное окно с уведомлением о временном техническом сбое на платформе.
* **Текст сообщения для пользователя:**
  > **Временный сбой файлового хранилища**
  > К сожалению, сервис хранения документов сейчас недоступен по техническим причинам. Мы уже занимаемся восстановлением работы системы. Пожалуйста, повторите попытку загрузки через 2–3 минуты.

### ERR-06: Клиентская отмена загрузки
* **Системный поток:**
  1. Пользователь нажимает «Крестик» (Отмена) во время отправки бинарного тела в S3.
  2. Браузер обрывает текущую сессию передачи данных.
  3. Клиентское приложение шлёт запрос `DELETE /internal/v1/documents/uploads/{uploadId}` (либо сессия аннулируется автоматически по истечении TTL 2 часа).
  4. Сессия загрузки аннулируется; версия остаётся в `NEW` и в бизнес-логике не участвует (отдельного статуса отмены нет — MZD-33 §1). Воркер очистки абортирует незавершённые S3-multipart-загрузки и удаляет недогруженные части; пустая карточка без валидных версий удаляется из выдачи.
* **Реакция интерфейса (UI):** строка загрузки исчезает из списка, интерфейс возвращается в состояние выбора файла.
* **Текст сообщения для пользователя (в случае уведомления об очистке):**
  > **Загрузка отменена пользователем**
  > Передача файла `{fileName}` была успешно прервана. Временные данные удалены. Вы можете выбрать другой файл для загрузки в любое время.
