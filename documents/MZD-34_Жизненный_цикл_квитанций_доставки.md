# MZD-34: Жизненный цикл квитанций доставки (Delivery Receipts)

Настоящая спецификация определяет логику механизма гарантированного отслеживания доставки и ознакомления заявителя с опубликованными документами через реестр Личного Кабинета (ЛК). Факты доставки — **не статусы карточки документа**, а иммутабельные записи `DeliveryReceipt` (SPEC-13 §1.3); статус карточки при их регистрации остаётся `PUBLISHED`.

## 1. Реестр типов квитанций (`ReceiptType`, SPEC-13 §1.3)

1. **Доставлен (`DELIVERED`)**: регистрируется автоматически в момент, когда API-шлюз ЛК возвращает список документов для рендеринга и метаданные конкретного документа успешно отобразились у пользователя.
2. **Открыт (`OPENED`)**: фиксируется, когда пользователь кликает на карточку документа в ЛК, переходя на экран детальной информации или запуская встроенный просмотрщик.
3. **Скачан (`DOWNLOADED`)**: регистрируется в момент успешного обращения пользователя за временной presigned-ссылкой для сохранения файла на устройство.

Факт формирует `cabinet-service`, хранит `document-service`. Регистрация квитанции публикует событие `DocumentDeliveryReceiptRegistered` (MZD-40). Перечень закрыт формулировкой REQ-08-011 и расширению не подлежит.

**Коды ошибок:** квитанция по документу не в статусе `PUBLISHED` → `409 Conflict`; `receiptType` вне допустимого набора → `422 Unprocessable Entity` (MZD-25). Для `SERVICE_RESOLUTION` квитанции невозможны в принципе — документ не публикуется (MZD-46, F1.7).

## 2. Ключ защиты от дублирования событий (Idempotency Key)

Для исключения повторной регистрации одинаковых статусов доставки при множественных кликах, перезагрузках страниц или retry-запросах фронтенда применяется строгий композитный ключ уникальности.

* **Состав ключа:** `unique_delivery_event_idx` = `[document_id, recipient_id, receipt_type, publication_id]`, где `publication_id` — идентификатор акта публикации (генерируется при каждом успешном `/publish`).
  * Включение `publication_id` обязательно: без него при повторной публикации документа (штатный сценарий исправления ошибки — новая версия + повторный `/publish`, Q-5) новые квитанции невозможно было бы записать — старые записи иммутабельны и навсегда занимали бы ключ `[document_id, recipient_id, receipt_type]`.
* **Логика обработки на уровне БД:** при попытке повторной записи события с той же комбинацией полей (например, повторный клик на скачивание того же файла тем же пользователем) вставка игнорируется через `ON CONFLICT DO NOTHING`, клиенту возвращается `200 OK` без генерации новой записи.
* **Направление переходов:** фиксация событий строго линейна в рамках одного акта публикации. События `OPENED` и `DOWNLOADED` не могут быть зарегистрированы, если для данной связки `[document_id, recipient_id, publication_id]` ещё нет записи `DELIVERED`.

## 3. Исправление ошибочной публикации и повторная публикация

**Отзыв публикации в MVP не реализуется** (Q-5, close-questions): соответствующего REQ в ТЗ нет, обратного перехода из `PUBLISHED` в графе MZD-22 нет — функция в backlog (для неё в SPEC-13 §3.3 зарезервирован `reasonCode = PUBLICATION_REVOKED`).

* **Штатный сценарий исправления ошибки:** загрузка новой версии файла и повторная публикация (`/publish`). Зафиксированный риск: отозвать уже выданный документ из ЛК в MVP нельзя; смягчение — короткий TTL защищённых ссылок (15 минут по умолчанию, MZD-36), вечного доступа к файлу у заявителя нет.
* **Принцип иммутабельности данных:** квитанции — строгий юридический и технический аудит. Они **никогда не удаляются** и не изменяются, даже если документ был опубликован ошибочно.
* **Поведение при повторной публикации:**
  1. Генерируется новый `publication_id`, и жизненный цикл отслеживания (`DELIVERED → OPENED → DOWNLOADED`) начинается заново.
  2. Все ранее сгенерированные квитанции остаются в историческом реестре аудита со своими исходными временными метками и исходным `publication_id` — для расследования инцидентов и подтверждения факта ознакомления на определённый момент времени.

## 4. Правило абсолютного сохранения истории (WORM для квитанций)

* **Запрет на операции удаления и изменения:** для таблицы `delivery_receipt` на уровне СУБД заблокированы любые операции `DELETE` и `UPDATE`. Права на данные действия отсутствуют у сервисного аккаунта `document-service` в промышленной среде.
* **Режим работы с данными:** допускаются только `INSERT` (добавление факта) и `SELECT` (чтение лога доставки для аналитики и АРМ ведомства).
