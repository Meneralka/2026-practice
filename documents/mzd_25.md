# Матрица критериев приёмки (контракт-тестопригодная)

**Документ:** вход в SPEC-11
**Целевой сервис:** `document-service`
**Источники доменов:** SPEC-04 (@koshak1030), SPEC-09 (@ArtemKvvs) — оба прошли ревью Denis. Сквозные: MZD-22 (стейт-машины), SPEC-14 (envelope), реестр REQ-08 (ТЗ р.8), ТЗ р.10, ФТ, НФТ.
**Статус:** критерии переложены в тестопригодный вид; домены не сочинены QA.

---

## Легенда

- **SCC** — проверяемо Spring Cloud Contract (граница сервиса: HTTP-стаб / Kafka-message; структура, коды, схема события).
- **INT** — проверяемо смежным интеграционным / `@SpringBootTest`-тестом (внутреннее состояние, недоступное SCC).
- **Приоритет** — MUST (блокирует релиз) / SHOULD (тестируется, не блокирует).

**Единый код на кейс (зафиксировано):**

| Кейс | Код |
|---|---|
| Неразрешённый MIME | 415 |
| Мутация существующей версии | 409 |
| Операция над карантинным документом | 409 |
| Конфликт `idempotencyKey` (тот же ключ, другое тело) | 409 |
| Истёкшая ссылка | 410 |
| Несуществующий документ/цель/карточка | 404 |
| Нарушение бизнес-правила (нет обяз. связи, недопустимый `visibilityProfile`, недопустимый `receiptType`, размер > лимита, ttl > max) | 422 |
| Отказ доступа (чужой документ/организация, guard SERVICE_RESOLUTION, действие не-участником маршрута) | 403 |
| Недоступность смежного сервиса | 503 |

**Статусы документа (MZD-22 / SPEC-13):** `DRAFT`, `UNDER_REVIEW`, `ACCEPTED`, `REPLACEMENT_REQUIRED`, `UNDER_APPROVAL`, `APPROVED`, `PUBLISHED`, `ARCHIVED`.
**Статус файла (ФТ-07, MZD-16):** `NEW`, `UPLOADED`, `CHECKING`, `CLEAN`, `INFECTED`, `INVALID`.
**Kafka-топология:** топик-на-сервис-продюсер (`document.events.v1`, `request.events.v1`, …).

## A. Домен документов/доставки (SPEC-04, @koshak1030)

### A1. Двухфазная загрузка (init → complete)

| Поле | Значение |
|---|---|
| REQ / ФТ | SPEC-04 · REQ-08-001 (множественная загрузка — см. C1) · **MUST** |
| Позитив | `init` с валидным контекстом → 200 + `uploadId`, `documentId`, `objectStorageTarget`, `maxSize`, `allowedMimeTypes`; `complete` → 200 + `fileVersionId`, карточка в `DRAFT` |
| Негатив | `complete` с несуществующим/чужим `uploadId` → 404; повтор `complete` того же `uploadId` → идемпотентный тот же ответ (не вторая версия) |
| SCC | структура ответов `init`/`complete`, коды 200/404, идемпотентный ответ на повтор |
| INT | что `complete` создал ровно одну `FileVersion` |

### A2. Whitelist MIME (Apache Tika)

| Поле | Значение |
|---|---|
| REQ / ФТ | SPEC-04 · **REQ-08-002** (техническая и антивирусная проверка) · **MUST** |
| Позитив | тип из `allowedMimeTypes` (напр. `application/pdf`) → загрузка проходит |
| Негатив | тип вне whitelist или подменённый (расширение .pdf, Tika определяет иное) → **415**, версия не сохраняется |
| SCC | код 415 и тело ошибки на неразрешённый MIME |
| INT | что Tika определяет тип по содержимому, а не по расширению (подмена ловится) |

### A3. Лимит размера файла

| Поле | Значение |
|---|---|
| REQ / ФТ | SPEC-04 · REQ-08-001/002 · **MUST** |
| Позитив | размер ≤ **50 МБ** → проходит; отсечка задаётся `content-length-range` в presigned-политике |
| Негатив | размер > 50 МБ → **422**; объект удаляется из хранилища; на `complete` дополнительно `HEAD` проверяет фактический размер |
| SCC | код 422 и сообщение при превышении на `complete` |
| INT | отсечка presigned-политикой (`content-length-range`) + `HEAD`-проверка на `complete`; фактическое удаление объекта |

### A4. Карантин при провале антивируса (ФТ-07)

| Поле | Значение |
|---|---|
| REQ / ФТ | SPEC-04 · **REQ-08-002** · **ФТ-07** (MZD-22) · **MUST** |
| Позитив | `FileVersion.status=CLEAN` → карточка выходит из `DRAFT`, бизнес-переходы и связи допустимы |
| Негатив | `FileVersion.status=INFECTED` → карточка остаётся в `DRAFT`, `DocumentLink` **не создаётся**, переход в `UNDER_REVIEW` запрещён; любая операция над карантинным документом (создание связи/смена статуса) → **409** |
| SCC | отказ `links`/`status`-endpoint кодом 409 для документа с файлом не в `CLEAN` |
| INT | что при `INFECTED` карточка не покинула `DRAFT` и связи физически отсутствуют; переход на бизнес-стадии только для `CLEAN` |

### A5. Версии без перезаписи (WORM)

| Поле | Значение |
|---|---|
| REQ / ФТ | SPEC-04 · **REQ-08-004** (версии без перезаписи) · **ФТ-05** (целостность) · **MUST** |
| Позитив | новая версия → `DocumentVersionCreated` (`versionNumber+1`, `previousVersionId` заполнен), старая версия читаема |
| Негатив | попытка перезаписать/удалить существующую `fileVersionId` → **409**; object lock не даёт мутацию |
| SCC | схему `DocumentVersionCreated` (наличие `previousVersionId`), код 409 на мутацию версии |
| INT | что старый объект в хранилище неизменен (object lock / WORM) |

### A6. TTL secure-link

| Поле | Значение |
|---|---|
| REQ / ФТ | SPEC-04 · **REQ-08-009** (защищённые временные ссылки) · **SHOULD** |
| Позитив | `createSecureLink(action)` → 200 + `secureLinkRef`, `expiresAt` = now + **TTL по умолчанию 15 мин** (задаёт сервер); клиент может запросить ttl вплоть до **max 60 мин** |
| Негатив 1 | доступ по ссылке после `expiresAt` → **410** |
| Негатив 2 | клиент запросил `ttl > 60 мин` → **422** |
| SCC | ответ создания ссылки, наличие `expiresAt`, код 410 на истёкшую, 422 на превышение max ttl |
| INT | фактическое истечение по времени; что TTL проставляет сервер, а не клиент |

### A7. Публикация заявителю с проверкой visibilityProfile

| Поле | Значение |
|---|---|
| REQ / ФТ | SPEC-04 · **REQ-08-010** (публикация документа заявителю) · MZD-22 (граф `PUBLISHED` по типу) · **MUST** |
| Позитив | `publish` документа в `APPROVED` с валидным `visibilityProfile` → 200 + `DocumentPublished` (`applicantId`, `visibilityProfile`, `secureLinkRef`), статус → `PUBLISHED`; заявитель видит результат в ЛК |
| Негатив | publish документа не в `APPROVED` → **409**; publish типа, где `PUBLISHED` исключён графом (напр. `SERVICE_RESOLUTION`, `REQUEST_MATERIAL`) или недопустимый `visibilityProfile` → **422** |
| SCC | коды 200/409/422, схему `DocumentPublished` |
| INT | связку «статус=`APPROVED`» + допустимость `PUBLISHED` по типу (MZD-22); отображение в ЛК |

### A8. Delivery receipt

| Поле | Значение |
|---|---|
| REQ / ФТ | SPEC-04 · **REQ-08-011** (delivery receipt: доставлен/открыт/скачан) · **SHOULD** |
| Позитив | регистрация `delivered/opened/downloaded` → `DocumentDeliveryReceiptRegistered` (`receiptType`, `occurredAt`) |
| Негатив | `receiptType` вне допустимого набора → **422**; receipt по документу не в `PUBLISHED` → **409** |
| SCC | схему события/ответа, допустимые значения `receiptType`, коды 422/409 |
| INT | переходы состояний receipt |

## B. Интеграционный домен (SPEC-09, @ArtemKvvs)

### B1. Регистрационные журналы и нумераторы — gaps allowed (ФТ-29)

| Поле | Значение |
|---|---|
| REQ / ФТ | SPEC-09 · **REQ-08-006** (регистрационные журналы и нумераторы) · **ФТ-29** (gaps allowed, обоснование производительности) · **MUST** |
| Позитив | присвоение номера при регистрации → номер **уникален**; при откате транзакции номер **сгорает** (разрыв допустим и не является дефектом); повтор с тем же `idempotencyKey` возвращает **тот же** номер |
| Негатив | два разных запроса → нет **дубля** номера; разрыв нумерации после отката НЕ считается ошибкой (не проверяется как дефект) |
| SCC | уникальность/формат номера в ответе регистрации; тот же номер на повтор `idempotencyKey` |
| INT | отсутствие дублей под конкурентной нагрузкой; сгорание номера при откате не блокирует выдачу следующих |

### B2. Идемпотентность (idempotencyKey)

| Поле | Значение |
|---|---|
| REQ / ФТ | SPEC-09 · сквозное для команд REQ-08-001/005/007/010 · **MUST** |
| Позитив | команда с `idempotencyKey` → 200; повтор с тем же ключом → тот же результат, без второго эффекта |
| Негатив | тот же `idempotencyKey` с другим телом → **409**; неидемпотентный POST без ключа не ретраится |
| SCC | идентичность ответа на повтор ключа, код 409 на конфликт ключ/тело |
| INT | отсутствие побочного эффекта (одна запись, одно событие) |

### B3. Outbox

| Поле | Значение |
|---|---|
| REQ / ФТ | SPEC-09 · сквозное для REQ-08-012 (события) · **MUST** |
| Позитив | успешная транзакция изменения документа → событие гарантированно попадает в `document.events.v1` |
| Негатив | откат транзакции → событие **не** публикуется (нет фантомных событий) |
| SCC | схему публикуемого события (producer verifier) |
| INT | атомарность «запись + outbox», отсутствие потери и фантома |

### B4. DLQ / replay — 5 попыток

| Поле | Значение |
|---|---|
| REQ / ФТ | SPEC-09 · стейкхолдер-документ р.10 (5 попыток) · **MUST** |
| Позитив | обработка успешна в пределах 5 попыток → без DLQ |
| Негатив | после исчерпания **5** попыток → сообщение в DLQ с `errorClass`, `retryCount`, `correlationId`; replay после устранения причины → успешная обработка без дубля |
| SCC | — (DLQ — внутренний топик, контракта с потребителем нет) |
| INT | схема DLQ-сообщения; сам механизм retry → DLQ → replay (на 5-й неудаче уходит в DLQ) |

### B5. Обязательность DocumentLink

| Поле | Значение |
|---|---|
| REQ / ФТ | SPEC-09 · **REQ-08-005** (связь документа) · **MUST** |
| Позитив | регистрация сравнительного протокола **с** `linkType=PRIMARY_EXPERTISE` → 200 |
| Негатив | регистрация протокола **без** обязательной связи → **422** |
| SCC | код 422 на отсутствие обязательной связи, 200 при наличии |
| INT | матрицу обязательности связей по типам документов |

## C. Функциональные требования ТЗ р.8 (REQ-08-xxx)

### C1. REQ-08-001 — Множественная загрузка (MUST)

| Поле | Значение |
|---|---|
| REQ / ФТ | **REQ-08-001** · **MUST** |
| Позитив | пакет из N файлов → каждый обрабатывается независимо; успешные создают версии; действие фиксируется в журнале |
| Негатив | ошибка по одному файлу (INFECTED / размер > 50 МБ / MIME) → откатывается **только он**, остальные проходят; ответ содержит по-файловый статус |
| SCC | структура batch-ответа с per-file статусами и кодами |
| INT | что провал одного файла не откатывает транзакции остальных (частичный успех); запись в журнал действий |

### C2. REQ-08-003 — Карточка документа и реквизиты (MUST)

| Поле | Значение |
|---|---|
| REQ / ФТ | **REQ-08-003** · **MUST** |
| Позитив | `GET /documents/{id}` → полнота реквизитов (`documentType`, `status`, `versionNumber`, `links`, `visibilityProfile`), `actualFileVersionId` указывает на **актуальную** версию |
| Негатив | несуществующий `documentId` → **404**; без прав → **403** |
| SCC | схему `DocumentCard`, полноту обязательных полей, коды 200/404/403 |
| INT | что `actualFileVersionId` = последней версии после серии загрузок |

### C3. REQ-08-005 — Связь документа (M:N) + идемпотентность (MUST)

| Поле | Значение |
|---|---|
| REQ / ФТ | **REQ-08-005** (связь с заявкой/делом/картой/отчётом/маршрутом) · **MUST** |
| Позитив | создание связи → 200 + `DocumentLinkCreated`; один документ ↔️ много целей и одна цель ↔️ много документов (M:N); повтор той же связи → идемпотентно тот же результат |
| Негатив | связь на несуществующую цель → **404**; повтор существующей связи не создаёт дубль и не является ошибкой |
| SCC | коды создания/повтора, схему `DocumentLinkCreated`, идемпотентность |
| INT | M:N-кардинальность в хранилище; отсутствие дублей связей |

### C4. REQ-08-007 — Маршрут согласования и утверждения (MUST)

| Поле | Значение |
|---|---|
| REQ / ФТ | **REQ-08-007** · **MUST** |
| Позитив | `startDocumentRoute` → маршрут запущен + `DocumentApprovalRequested`, статус → `UNDER_APPROVAL`; `completeRouteTask(decision=APPROVE)` → `DocumentApproved`, статус → `APPROVED` |
| Негатив | `completeRouteTask` не-участником маршрута → **403**; отклонение согласующим → `DocumentRejectedByApprover` + переход в `REPLACEMENT_REQUIRED` (где допустим графом); повтор завершённой задачи → идемпотентно |
| SCC | коды, схемы `DocumentApprovalRequested`/`DocumentApproved`/`DocumentRejectedByApprover` |
| INT | переходы стейт-машины при approve/reject по MZD-22 |

### C5. REQ-08-008 — Предпросмотр файлов в браузере (MUST)

| Поле | Значение |
|---|---|
| REQ / ФТ | **REQ-08-008** · **MUST** |
| Статус | ⚠️ **ПРОБЕЛ: контракт предпросмотра не определён.** Механика (inline-рендер / конвертация / стриминг из хранилища), endpoint и формат ответа отсутствуют в SPEC-04/09, MZD-22, стейкхолдер-документе. Требование MUST |
| Позитив | `[требует определения контракта — получить у @koshak1030]`: функция доступна через UI/API, тестовый сценарий, фиксация в журнале |
| Негатив | `[требует определения контракта]` |
| Действие | вынести в запрос бэкендеру: endpoint предпросмотра, формат, права доступа, TTL. Без этого MUST-функция не покрывается тестом |

### C6. REQ-08-012 — Envelope события (SPEC-14) (MUST)

| Поле | Значение |
|---|---|
| REQ / ФТ | **REQ-08-012** (события DocumentUploaded/Accepted/Published) · **SPEC-14** · **MUST** |
| Позитив | любое публикуемое событие содержит полный envelope: `eventId`, `eventType`, `eventVersion`, `correlationId`, `causationId`, `businessKey`, `payload`; корректно обрабатывается потребителем |
| Негатив | событие без `correlationId` или с неизвестной major-версией `eventVersion` → consumer отклоняет / не обрабатывает |
| SCC | схему envelope на producer-verifier (наличие всех 7 полей, тип `eventVersion`) |
| INT | проброс `correlationId`/`causationId` по цепочке REST → Kafka → consumer |

## D. Сквозные требования

### D1. ФТ-05 — SHA-256 и целостность версии

| Поле | Значение |
|---|---|
| REQ / ФТ | **ФТ-05** · связано с REQ-08-004 · **MUST** |
| Позитив | `complete` → `sha256` посчитан и сохранён в `FileVersion`; повторное чтение даёт тот же хэш |
| Негатив | несовпадение фактического содержимого и `sha256` (повреждение/подмена) → версия помечается невалидной, не публикуется |
| SCC | наличие `sha256` в ответе/событии версии |
| INT | фактический пересчёт SHA-256 и детект расхождения (целостность) |

### D2. MZD-22 — SERVICE_RESOLUTION: запрет транзитов

| Поле | Значение |
|---|---|
| REQ / ФТ | **MZD-22** (жёсткий guard) · **MUST** |
| Позитив | `SERVICE_RESOLUTION` проходит только `DRAFT → UNDER_APPROVAL → APPROVED → ARCHIVED` |
| Негатив | попытка транзита в `UNDER_REVIEW`/`ACCEPTED`/`PUBLISHED`/`REPLACEMENT_REQUIRED` → guard → `SecurityAccessDeniedException` → **403**; метаданные `SERVICE_RESOLUTION` **не уходят в ЛК заявителя** |
| SCC | код 403 на запрещённый транзит |
| INT | безусловный guard в конфигурации автомата; отсутствие утечки метаданных в кабинет |

### D3. ТЗ р.10 — RBAC / ABAC

| Поле | Значение |
|---|---|
| REQ / ФТ | **ТЗ р.10** · **MUST** |
| Позитив | доступ к своему документу в рамках своей организации/прав → 200 |
| Негатив | чужой документ → **403**; документ чужой организации → **403** (проверка на gateway и в сервисе — defense in depth) |
| SCC | код 403 при отсутствии прав; проверка присутствия/валидации JWT |
| INT | ABAC-логику (принадлежность организации, `accessScope`, `visibilityProfile`) |

### D4. DoD — Аудит действий (журнал)

| Поле | Значение |
|---|---|
| REQ / ФТ | **DoD (сквозное)** · требование «фиксируется в журнале действий» для всех REQ-08 · **MUST** |
| Позитив | каждое изменяющее состояние действие пишет запись в журнал: `actor`, `action`, `changedAt`, `correlationId` |
| Негатив | — (аудит — write-side; отсутствие записи выявляется INT) |
| SCC | — (персистентность аудита вне границы контракта) |
| INT | что действие (загрузка, смена статуса, публикация, связывание) породило корректную запись в журнал действий |

### D5. НФТ — Отказоустойчивость

| Поле | Значение |
|---|---|
| REQ / ФТ | **НФТ** (timeout / circuit breaker / недоступность смежных) · **MUST** |
| Позитив | синхронный вызов в пределах таймаута (connect 2 c / response 5 c) → ответ |
| Негатив | смежный сервис недоступен → **503** клиенту; circuit breaker размыкается при ≥50% ошибок на окне ≥20 вызовов; идемпотентные операции ретраятся (до 3 попыток, backoff 200 мс / 1 c / 3 c); неидемпотентные без ключа — нет |
| SCC | форму ответа 503 при недоступности |
| INT | сам circuit breaker, retry-политику, fallback на read-model |
